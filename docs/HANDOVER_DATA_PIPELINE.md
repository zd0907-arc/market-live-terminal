# 数据管道与底层架构交接白皮书 (Data Pipeline Handover)

> **文档目的**：本文档是在完成 `Tailscale` 跨网直连后，由研发与 PM 进行的一次战略级验收与架构梳理。在此记录当前系统的状态、已发现的业务逻辑瓶颈、以及未来处理“历史 L2 与 实时数据”融合的技术路线图。

---

## 核心焦点一：大单/超大单阈值的动态对齐

**当前问题**：
`etl_worker_win.py` 等清洗脚本目前使用的是硬编码默认值（主力 $\ge$ 20万，超大单 $\ge$ 100万）。如果前端用户在 Web 页面上修改了这套 `app_config`，离线数据和实时聚合数据将产生标准上的撕裂。

**解决方案（未来计划）**：
1. **摒弃离线硬编码**：Windows 离线跑历史包时，不能自作主张。它应该先通过 API 或查库的方式，拉取系统当前的 `app_config`。
2. **底层原始表保留**：为了支持修改配置后的**追溯重算**，系统绝对不能只存 `local_history`（日级别汇总）或 `history_30m`。**必须毫无保留地存储每一笔最原始的 `trade_ticks`。**
3. **动态重算流水线 (Re-Aggregation)**：只要阈值发生变更，云端的 Python Job 需要被触发。它将基于底层没被破坏的几十亿条 `trade_ticks`，根据新的界线重新执行 Bucket 求和，刷新出新维度的 `local_history` 表。

---

## 核心焦点二：拆单还原逻辑 (L2 深度识别) 的暂缓执行

**当前的业务两难**：
*   **历史数据 (L2)**：我们花钱购买的压缩包里，带有及其宝贵的 `BuyOrderID` 和 `SaleOrderID`。如果要精准还原机构的“一单拆百笔”，必须按 ID 进行 Pandas 的 `groupby` 求和运算。
*   **实时数据 (AkShare接口)**：这是白嫖的实时公开接口，它**只提供单笔撮合明细，不提供背后的逻辑单号 OrderID**。

**PM 决策结论**：**暂不执行历史 L2 的拆单还原**。
如果我们在 `2025.1-2026.2` 这一年半的历史数据中启用了真实的 OrderID 合并，而在今天的实时图表里依然是用白嫖的低级碎片拼接，这将导致昨天和今天的核心数据产生断层（例如昨天的日内大单占比 40%，今天因为散单无法合并突然降到 5%）。为了保证模型输入的**“口径一致性”（Consistency）**，我们目前的历史脚本将继续保持“基于单笔总额”而非“基于单号”的妥协方案。

该痛点将明确预留，等待下文【焦点三】的付费实时数据接入后一并解决。

---

## 核心焦点三：目标表结构与底层数据流域总览

要实现这个庞大金融终端的“永动机”，我们需要理清现在到底有哪几种表，谁灌给谁。

### 数据流向图 (Data Flow)

| 存储层级 | 表名 / 用途 | 写入频次 / 来源角色 |
| :--- | :--- | :--- |
| **【Raw 源表】** | `trade_ticks` (逐笔明细表) | 极高频。白天由 Windows 雷达实时 POST；历史由压缩包写入。是万物之源，必须无限堆积保留。 |
| **【Raw 源表】** | `sentiment_snapshots` (情绪盘口表) | 极高频（3秒）。白天由 Windows 雷达实时快照插入。用于展示盘中分时多空比。 |
| **【Raw 源表】** | `sentiment_comments` (散户论坛表) | 中高频。云端自带爬虫去爬取股吧等舆情。 |
| **【衍生表】** | `history_30m` (半小时 K 线表) | 原则上应该基于 `trade_ticks` 的 30分钟切割汇总。目前图快，是通过 AkShare/压缩包 直接算好怼入。 |
| **【衍生表】** | `local_history` (日级主沉淀表) | 包含 `activity_ratio`, `net_inflow` 等终极指标。每天盘后基于前两种表计算生成，用于复盘和顶层大图展示。 |
| **【衍生表】** | `sentiment_summaries` (AI报告表) | 低频。利用大模型吃掉舆情后生成的精华总结。 |

### 面向未来的扩容：接入实时付费 L2 数据源
当未来我们决定花钱购买高质量的实时 L2 串流（比如直接接同花顺、万得或者券商的高频 WebSocket）时，架构应如下演进：

1. **替换 雷达之窗 (Windows)**：Windows 上现在的 `live_crawler_win.py` 将被重写。它不再拉取 AkShare，而是直连付费的 WebSocket 流。
2. **数据源质变**：传给腾讯云服务器的数据中，将正式包含 `BuyOrderID` 和 `SaleOrderID`。
3. **解封历史实力**：此时，我们不仅可以在实时流中运用真实的单号还原主力意图；还可以把曾经尘封的 2025.1-2026.2 压缩包重新翻出来，打上补丁进行一次“真正的全量 GROUP BY 洗礼”。
4. **底层平滑过渡**：由于我们现在一直死死守住 `trade_ticks` 这张最底层的源头表不删除，未来的升级对图表层是**完全向上兼容**的。这就是我们现在架构的底气。

## 总结：下一步研发规划 (Current Roadmap)

*   **当前已完成**：Mac <-> Windows 跨网直连自动化 (`sync_to_windows.sh`)；梳理并冻结 `etl_worker_win.py` 的改造方向（为保持前后口径一致，暂不强行做 groupby 拆单）。
*   **短期 Next Steps**：处理 `etl_worker_win.py` 在表名 (`history_daily` => `local_history`) 上的致命 Bug，确保大清洗脚本执行时不会因为云端表结构变更而彻底崩溃。
*   **长期 Next Steps**：将硬编码（20万/100万）抽象，开发 `config_sync` 同步机制。升级实时高频 L2 接口，最后全量重开 L2 订单聚合引擎。
