# 前后端双核 AI 协同机制 (AI Handoff Protocol)

## 0. 设计初衷
本项目开始进入“双核 AI 共建”时代：
- **前端 AI**：专注 React、TS、Vite 构建、可视化图表及交互体验（目前您正在与前端 AI 对话）。
- **后端 AI**：专注 Python FastAPI 架构、SQLite 性能优化、清洗脚本及底层数据路由。

由于两个 AI 的会话上下文互相隔离，为了避免认知错位、代码互相踩踏或改写结构断片，必须通过本文档和一套“硬盘级”的协议文本建立物理通讯。

---

## 1. 唯一契约源 (Single Source of Truth)
> **原则：契约驱动开发 (Contract-Driven Development)**

`docs/03_DATA_CONTRACTS.md` 是前后端的唯一法律文件。
- **发起方责任**：无论是前端 AI 需要新数据，还是后端 AI 重构了字段，**动手写代码前，必须先更新 `03_DATA_CONTRACTS.md`。**
- **接收方责任**：任何 AI 接到任务时，如果不确定 API 怎么获取，第一步是 `cat/view` 这个契约文件。以文档定义的结构为准，不准凭空“猜想” API 的入参出参。

---

## 2. 硬隔离边界与禁区
为了确保不把对方拉下水，请双方遵守活动边界：
- **前端 AI 隔离区**：工作空间在 `src/` 目录、`index.html`。绝不能修改 `backend/` 目录内任何 Python 架构。需要新接口时，只写 Mock 代码并留言给后端。
- **后端 AI 隔离区**：工作空间在 `backend/` 目录、`data/` 或 Python 脚本配置。绝不要改动 React 的渲染树结构。

---

## 3. 标准通讯管道：交接看板 (AI Handoff Log)
在项目库中建立 `docs/AI_HANDOFF_LOG.md` 作为“换班日记本”。

**【强制动作 SOP】**：
- **每次开发完毕，准备休息时（或用户主动切换 AI 前）**：当前工作 AI 必须打开 `AI_HANDOFF_LOG.md`，使用 Append 模式在开头追加本次的【工作简报】和【需对方配合的 TODO】。
- **每次新开会话，初次见到用户时**：AI 必须使用工具优先读取 `docs/AI_HANDOFF_LOG.md`，看看上一个哥们儿/自己上一次留下的坑和任务指令。

**日志模板格式**：
```markdown
## [时间戳] [前端 AI / 后端 AI] 阶段代号
**1. 完成了什么？**
- 简述核心变动，修改了哪几个文件。
**2. 变更了契约吗？**
- 指明是否有更新 `03_DATA_CONTRACTS.md`。
**3. 接下来需要另一个 AI 做什么？ (To-Do for Next)**
- a. 后端 AI 兄弟，我已经加了 xxx 的 UI 按钮，点击会向 `/api/xxx` 发 POST 请求。你按契约把它实现了就行。
```
