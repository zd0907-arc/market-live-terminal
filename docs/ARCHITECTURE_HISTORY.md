# 架构演进与决策记录 (Architecture Decision Records - ADR)

本文档旨在记录 `ZhangData` 项目诞生以来的**关键架构变迁史 (Why things changed)**。
这是系统保持防呆与免疫力的核心文件。**对未来的所有 AI 贡献者：在计划修改任何架构前，请务必阅读本文档，并强制更新你要做出的任何架构变动。**

---

## [ADR-001] 从纯前端转向前后端分离架构
- **日期**: 2025 年末
- **背景**: 浏览器存在严格的跨域限制 (CORS)，且前端无法存储持久化结构数据，导致一旦刷新图表流失。
- **决策**: 引入 FastAPI (Python) 作为后端微服务，承担向新浪、东财等数据源发起请求的“网关”角色，并将清洗后的高密度逐笔数据沉淀至 SQLite。
- **状态**: 已实施 (v2.0)

---

## [ADR-002] 引入离线算力网关 (Windows Node ETL) 规避 IP 封锁
- **日期**: 2026-02-28 (v4.0 时代)
- **背景**: 
  - 核心痛点：项目需回填长达数月乃至一年的历史盘口数据供 AI 和用户研究（涉及数十至数百GB的行情压缩包）。
  - 若在云端（腾讯云）服务器上直连公共金融 API 高频拉取历史全量数据，极易触发 API 防火墙导致服务器 IP 被永久封禁。
- **决策**: 
  - 本地 Mac 不做脏活；云端专心对外提供访问与小规模实时数据爬取。
  - 在局域网中引入并指定一台 Windows 台式机作为“离线算力提纯机”。将繁重的历史数据解压、遍历、聚合运算全部下放到 Windows 端执行。最后通过 `scp` 将提纯完毕的 SQLite 文件直接部署回云端主库 (`market_data.db`)。
- **收益**: 彻底隔离脏数据，云端 IP 绝对安全。
- **状态**: 已实施 (v4.0 分布式 ETL)

---

## [ADR-003] 本地共享云端快照原则 (Single Source of Truth)
- **日期**: 2026-02-28
- **背景**: 上述架构实施后，出现了“云端是全量数据，而 Mac 本地开发测试环境数据为空”的撕裂情况。这导致前端部分图表因无数据而频繁崩溃或隐藏，出现双胞胎数据库(Split-Brain)的路径错乱问题。
- **决策**:
  - 本地 Mac 绝不单独捏造测试数据，以免掩盖真实的边缘场景 Bug。
  - 引入 `sync_cloud_db.sh`，在开发测试前，开发者需一键将生产环境最真实的 `market_data.db` 全量拉下本地根目录，让本地 `localhost` 前端 100% 吃云端真实数据运行。
- **规范设立**: Python 环境下的 DB 路径全部重构为基于 `ROOT_DIR` 的强绝对路径。
- **状态**: 已实施
