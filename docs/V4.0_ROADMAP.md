# V4.0 系统精进与量化重构路线图 (Roadmap)

> **文档定位**：本文档是在 V4.0 分布式 ETL 架构（Windows 离线算力网关）确立后，结合多次 AI 深度复盘总结出的一份**阶段性冲刺指南**。
> 本文档为**过程性规划文件**，随着每个阶段的开发落地，本文档将被逐步划掉并在最终完成后归档至 `docs/archive/`。

---

## 一、 核心痛点与架构拍板 (The "Why")

在过去版本的迭代中，我们遭遇了以下四大设计陷阱，现在正式确立破解方案：

### 1. 阈值陷阱：“读时计算”导致的算力灾难
*   **现象**：前台提供随意修改“大单/超大单”阈值的入口。导致只要改一个数字，云端沉淀的全部 `local_history` 和 `history_30m` 瞬间失效，必须用 Windows 重新花几十个小时洗牌。
*   **终极决断**：**锁死口径，废除前端动态配置**。
    *   主力（大单）：单笔金额 $[20万, 100万)$
    *   超大单：单笔金额 $\ge 100万$
    *   这组参数由业务层定义为常量，绝不再随意变动。未来若真要改，必须走“大版本升级 + 数据全表重刷”的工程变更流程。

### 2. 界面分裂：“实时”与“历史”的物理割裂
*   **现象**：用户在盘中盯盘时看一套 UI，收盘复盘时又要切到另一个历史维度，导致组件无法复用，数据逻辑严重耦合。
*   **终极决断**：**大一统的时间序列图表 (Time-Series Chart)**。
    *   全面废除物理双页设计。采用**全局单页+时间轴控制条**（今日、近5日、自定义）。
    *   前端统一发一个请求 `/api/timeseries?range=1d`，由后端网关（BFF层）智能判断去查内存实时数据还是 SQLite 历史底表。做到“一个范围、一个粒度、全模块对齐”。

### 3. 散户情绪失真：“绝对数量”导致的误判
*   **现象**：盘中 10:30 发帖多，盘后 15:30 发帖少，导致情绪热度图永远是“早盘高，尾盘低”的无用抛物线。
*   **终极决断**：引入**相对基准热度 (Relative Heat Index)**。
    *   不再比拼绝对发帖量，而是计算**偏离度**（例如：对比过去 30 天同时段的平均发帖量，今天飙升了 400%）。
    *   取消孤立的柱状图，将平滑的“情绪相对热度面积图”硬刚对齐叠加在股价 K 线下方的相同时间轴上。

### 4. 冰山微观信号的孤岛化
*   **现象**：现在的 3 秒盘口快照（挂单）和逐笔成交（吃单）是分开展示的，必须依靠人脑在两个图表间连线推演。
*   **终极决断**：**“狙击手”视图 (Event Annotation)**。
    *   利用同轴优势，把算出来的“冰山压单”、“虚假撤单”以 🧊/👻 图标的形式，直接当作**Event（事件点）**打在 K 线上。

## 二、 分阶段落地规划 (Phased Execution Plan)

接下来的开发必须严格按照以下阶段按部就班地进行，不可越级。

### 阶段一：底层基建清理与“口径冻结” (破旧立新)
**目标**：彻底解决数据耦合，统一阈值基准。
*   **[ 后端 ]** 移除前端大单阈值配置的 API 依赖，将 $[20万/100万]$ 在 Python 计算代码里全面写死为全局常量。
*   **[ ETL ]** 修复此前 `etl_worker_win.py` 写入假表 (`history_daily`) 的 Bug，将表名对齐至标准的 `local_history` 并执行历史大清洗。
*   **[ 契约 ]** 落实“空态不静默”规范，确保所有没数据的日子返回清晰的 `message: "No data found"`，杜绝前端图表组件幽灵般消失。

### 阶段二：大一统数据网关与单页重构 (The Unified UI)
**目标**：消灭历史/实时隔离，一个入口包打天下。
*   **[ 后端 ]** 新增统一的 `/api/timeseries` 聚合 API。入参统一为 `start, end, granularity`。后端把原先散落的资金、K线、情绪，一次性齐胸包圆返回（数据结构保证 `fund_flow_series`, `sentiment_series` 长度和时间戳绝对对齐）。
*   **[ 前端 ]** 重写 UI，引入顶部时间拖拽轴。支持无缝滚动和缩放（DataZoom 组件融合）。首先接入基础的主图价格和底部的资金流向对比。

### 阶段三：情绪指标进化与微观狙击 (深度组合)
**目标**：让原本玄学的情绪变得严密可量化。
*   **[ 后端算法 ]** 把情绪库从“关键词命中数”升级为“基准时间段（如开盘前半小时历史均值）的偏离度打分”。
*   **[ UI 叠加 ]** 在统一前端的时间轴上，把情绪积分当作一个副图贴上去。把检测到的微观博弈冰山信号（`signals` 里的 JSON），化身为图标打点，点击可弹出当时买卖一的悬疑现场。

### 阶段四：量化沙盒构建 (The Screener & Backtester)
**目标**：激活辛苦洗好的 1 年历史数据，构建终极买卖研判闭环。
*   **[ 后端引擎 ]** 打造高阶 SQL/Pandas 选股引擎（例如：查找连续 3 天主力净流入且情绪热度处在冰点低谷的个股）。
*   **[ 模拟层 ]** 开发纯独立的回测沙盒：定义 Entry（进场）、Exit（出场），让这 1 年的全精度 L2 数据跑一遍，输出胜率/盈亏比大表。

### 阶段五（远景）：实盘量化前瞻 (向专业进化)
*   寻找或付费接入拥有原生真实 `OrderID` 的高质量机构级别 L2 流（WebSocket）。
*   一旦接通，将阶段四验证胜率最高的战法直接部署在腾讯云后端，盘中实时弹出狙击警告！
