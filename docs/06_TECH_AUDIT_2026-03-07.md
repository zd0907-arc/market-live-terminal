# 06_TECH_AUDIT_2026-03-07（技术总监级全局审查报告）

## 1. 审查范围与方法
- 审查范围：`backend/`、`src/`、`deploy/`、`docs/`、关键运维脚本。
- 审查方式：静态代码审查 + 运行验证（最小可执行命令）。
- 验证命令：
  - `python3 -m pytest -q backend/tests/test_core_algo.py`（当前结果：4 pass）
  - `npm run build`（结果：成功，但有大包与资源告警）
  - 鉴权护栏本地冒烟（`require_write_access` / `verify_token`）：未配置返回 503、错误 token 返回 401、正确 token 放行。

## 2. 管理层结论（Executive Summary）
- 当前项目可继续迭代，但**不具备“稳态生产系统”标准**。
- 主要风险集中在：
  1) 数据链路完整性（回填链路断裂、潜在写入错位）
  2) 生产安全（鉴权薄弱、密钥治理不足）
  3) 架构执行偏差（云端仍在主动外采，违反架构红线）
  4) 可测试性与可维护性（测试失效、文档契约漂移）

建议按 **P0 -> P1 -> P2** 三阶段治理；本报告中的 P0 条目已进入修复阶段，实时进展以 `docs/AI_HANDOFF_LOG.md` 最新记录为准。

## 3. 问题清单（按优先级）

### P0（必须立即处理，24-72小时）

#### F-001 自选股“历史回填”链路已断（功能表面存在，实则失败）
- 现象：添加自选时会尝试调用 `perform_historical_fetch`，但该函数不存在，后台任务直接异常。
- 证据：
  - `backend/app/routers/watchlist.py:28`
  - `backend/app/routers/watchlist.py:32`
- 影响：新加自选股的历史数据补齐不可靠，容易出现“刚加星标看不到历史/30m”的体验。
- 建议：
  1. 立即补齐函数实现或移除死调用。
  2. 给 `/api/watchlist` 增加后台任务状态回执（成功/失败原因）。

#### F-002 Backfill 写入字段顺序错误，存在数据错位风险
- 现象：backfill 组装 tick tuple 顺序与 `save_trade_ticks` 期望顺序不一致。
- 证据：
  - 构造顺序：`backend/app/services/backfill.py:215`
  - 落库调用：`backend/app/services/backfill.py:226`
  - 期望顺序定义：`backend/app/db/crud.py:58`
- 影响：一旦该逻辑被启用，`trade_ticks` 可能被写入错位字段（date/time/type错位），导致后续聚合异常。
- 建议：统一 tick DTO，改为显式命名字段映射，不允许裸 tuple 直传。

#### F-003 `/api/verify_realtime` 接口当前必定 500（响应模型与返回值不匹配）
- 现象：路由声明返回 `VerifyResult`，但实际返回字段 `sina_price/tencent_price/diff/status`。
- 证据：
  - `backend/app/routers/market.py:41`
  - `backend/app/routers/market.py:48`
  - `backend/app/models/schemas.py:28`
- 影响：调用该接口会产生运行时校验错误，破坏前后端契约可信度。
- 建议：要么下线该接口，要么修正响应模型和实现并补测试。

#### F-004 采集鉴权存在高风险（默认密钥 + 明文硬编码）
- 现象：ingest token 有默认值，Windows 启动脚本明文写死 token。
- 证据：
  - 默认 token：`backend/app/routers/ingest.py:16`
  - 明文 token：`start_live_crawler.bat:16`
- 影响：一旦仓库泄露或接口被探测，可伪造行情写入，直接污染生产数据。
- 建议：
  1. 禁止默认 token（未配置即拒绝启动 ingest）。
  2. 立即轮换密钥并迁移到安全注入（环境变量/密钥管理）。
  3. 增加来源 IP 白名单或签名机制。

#### F-005 生产写接口基本无鉴权，Nginx 对公网全开放
- 现象：`/api/watchlist`、`/api/config`、`/api/sentiment/crawl` 等关键接口无认证层。
- 证据：
  - `backend/app/routers/watchlist.py:17`
  - `backend/app/routers/config.py:42`
  - `backend/app/routers/sentiment.py:13`
  - `deploy/nginx.conf:21`
- 影响：任意访问者可改配置、刷爬虫、污染业务行为，形成运营与安全风险。
- 建议：引入最小可用认证（JWT/API Key + RBAC），并按接口分级授权。

#### F-006 代码实际行为违反架构红线（云端仍主动外采）
- 现象：云端启动后会开启 `collector`，并直接调用 AkShare。
- 证据：
  - 红线定义：`docs/01_SYSTEM_ARCHITECTURE.md:22`
  - 启动采集：`backend/app/main.py:85`
  - 外采调用：`backend/app/services/collector.py:64`
- 影响：与既定“云端只被动 ingest”原则冲突，导致架构行为不可预测且难排障。
- 建议：生产环境下彻底禁用云端外采（feature flag + 启动保护）。

### P1（高优先级，1-2周）

#### F-007 交易日历“工作日补齐”逻辑会误判节假日
- 现象：当上游日历不完整时，代码把缺失日期按周一到周五直接补成交易日。
- 证据：`backend/app/core/calendar.py:34`
- 影响：节假日可能被当作交易日，触发错误日期展示、错误任务调度。
- 建议：不要推断补齐；以权威交易所日历为准，并为“未知日期”返回不可交易。

#### F-008 实时接口计算开销高，存在扩容瓶颈
- 现象：每次请求都从 DB 拉全日 ticks，重建 pandas 并全量聚合；且保留 debug 级数据日志。
- 证据：
  - `backend/app/services/analysis.py:17`
  - `backend/app/services/analysis.py:28`
  - `backend/app/services/analysis.py:53`
- 影响：并发用户增长时 CPU 与 I/O 急剧上升，响应抖动明显。
- 建议：引入增量聚合缓存（按 symbol+date+last_tick_time），将全量重算改为增量更新。

#### F-009 多处线程“裸启动”且缺少并发治理
- 现象：路由中直接 `threading.Thread(...).start()`，缺少队列、上限、失败回收。
- 证据：
  - `backend/app/routers/watchlist.py:23`
  - `backend/app/routers/watchlist.py:31`
- 影响：高峰下线程失控、资源竞争与隐性失败概率上升。
- 建议：统一迁移到任务队列（如 APScheduler job queue / Celery / RQ）并做幂等。

#### F-010 数据契约文档与代码实现明显漂移
- 现象：契约写“raw层只追加、type=B/S/M、空状态统一 APIResponse、`/api/ticks_full` 存在”，实现已不一致。
- 证据：
  - 契约声明：`docs/03_DATA_CONTRACTS.md:13`
  - 契约字段约束：`docs/03_DATA_CONTRACTS.md:49`
  - 契约接口：`docs/03_DATA_CONTRACTS.md:102`
  - 空状态约束：`docs/03_DATA_CONTRACTS.md:111`
  - 实际覆盖写：`backend/app/routers/ingest.py:28`
  - 实际返回裸列表：`backend/app/routers/sentiment.py:275`
- 影响：前后端与多AI协作容易“按错契约开发”。
- 建议：把 `03_DATA_CONTRACTS.md` 拉回单一事实源，纳入 PR 必审。

#### F-011 测试体系失真（现有测试已红）
- 现象：测试引用不存在的方法 `check_spoof_buy`，当前无法全绿。
- 证据：
  - 测试调用：`backend/tests/test_core_algo.py:78`
  - monitor 中无该方法（算法段结束处）：`backend/app/services/monitor.py:228`
- 影响：CI 失去信号价值，回归风险无法前置拦截。
- 建议：先修复测试基线，再建立后端关键路径最小回归集（ingest、聚合、日期）。

### P2（中优先级，2-4周）

#### F-012 前端入口与产物治理存在历史残留
- 现象：`index.html` 同时存在 CDN Tailwind、importmap、本地入口与不存在的 `index.css` 引用。
- 证据：
  - `index.html:9`
  - `index.html:31`
  - `index.html:42`
- 影响：发布路径复杂、排障困难；`npm run build` 已出现资源告警。
- 建议：统一为 Vite 打包链路，移除 importmap/CDN 运行时依赖与死资源引用。

#### F-013 仓库存在第二份旧项目代码树，长期制造认知噪音
- 现象：根目录下嵌套 `market-live-terminal/`，版本显著落后。
- 证据：`market-live-terminal/package.json:4`
- 影响：AI/人类都容易误改错目录，构建上下文与代码审查成本上升。
- 建议：归档或删除旧代码树，仅保留单一工作副本。

#### F-014 README 与真实文档入口不一致
- 现象：README 指向多个当前不存在文档，且后端启动命令已过时。
- 证据：
  - 启动命令：`README.md:29`
  - 文档链接：`README.md:42`
- 影响：新成员上手成本高，误操作风险高。
- 建议：将 README 改成“5分钟可跑通”的最小可信手册。
