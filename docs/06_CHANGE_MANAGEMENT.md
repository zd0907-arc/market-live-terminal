# 06_CHANGE_MANAGEMENT（需求变更与阶段目标管理）

> 核心定位：定义“动态文档”如何创建、推进、收尾与归档。此文档是变更过程的唯一流程源。
>
> 边界提醒：业务规则看 `02`，接口契约看 `03`，发布执行看 `04`，协作规则看 `00`。

## 1. 目标
- 让需求、阶段目标、问题排查、配置调整都有统一落地载体。
- 保证执行顺序线性可追溯，不再“文档跳跃”。

## 2. 动态文档目录（非核心编号）
- 统一目录：`docs/changes/`
- 统一命名：`<TYPE>-YYYYMMDD-NN-<slug>.md`
- 当前新卡统一使用上述类型化编号；历史 `CHG-*` 只作为兼容引用保留，不再新建。
- `TYPE` 取值：
  - `MOD`：新模块
  - `REQ`：小需求
  - `INV`：问题排查
  - `CFG`：配置/数据调整
  - `STG`：阶段目标（版本阶段）

示例：
- `MOD-20260309-01-order-flow.md`
- `REQ-20260309-02-watchlist-sort.md`
- `INV-20260309-03-30m-gap-check.md`
- `CFG-20260309-04-write-token-rotate.md`
- `STG-20260309-01-v4-3-0-stability.md`

## 3. 单卡模板（所有 TYPE 通用）
1. 基本信息（ID/标题/状态/负责人/关联 Task ID/关联 CAP）
2. 背景与目标（为什么做）
3. 方案与边界（做什么/不做什么）
4. 执行步骤（按时间顺序）
5. 验收标准（Given/When/Then，绝对时间）
6. 风险与回滚
7. 结果回填（实际改动、验证结果、遗留问题）
8. 归档信息（归档时间、归档 ID、归档路径）

## 4. 执行顺序（强制，线性）
1. 在 `docs/changes/` 新建变更卡。
2. 在 `02` 对应 CAP 卡登记“拟变更点”。
3. 实施代码/配置改动并本地验证。
4. 如需发布，按 `04` 执行发布；生产冒烟由你手动执行，AI 负责给出清单与回填模板。
5. 发版任务必须执行版本一致性核对（`package.json`、`src/version.ts`、`README.md`）。
6. 回填 `02`（实现摘要+验收）与 `AI_HANDOFF_LOG`（短日志）。
7. 如有阻塞，更新 `07`；若阻塞已解决或需求方向已变，必须把对应项从 `07` 移除或归档。
8. 若 `AI_HANDOFF_LOG` 超出最近 `1~2` 个版本窗口，必须把旧窗口整理到 archive summary，再压缩主文件。
9. 任务完成后归档变更卡到 `docs/archive/changes/`（命名规则见 `docs/archive/ARCHIVE_NAMING_STANDARD.md`）。

## 4.1 两步快路径（默认推荐）
- **Step 1：文档管理**（`$governance-doc-keeper` AFTER）
  - 建/更变更卡 + 回填 `02` + 更新 `AI_HANDOFF_LOG`，状态置为 `RELEASE_READY`。
- **Step 2：发布执行**（`$release-ops-commander`）
  - 完成版本同步、部署、给出“业务冒烟清单（用户手动）”。
  - 发布后状态置为 `RELEASED_PENDING_SMOKE`，本轮到此结束，不再额外触发“二次发布”。
- **归档收口**
  - 你回传“业务冒烟通过”后，AI 仅做文档归档（`ARCHIVE`），**不触发再次部署**。

## 4.2 需求开发收口标准（当前正式规则）
1. 先在 `main` 只读排查范围，不直接改；
2. 一旦要改 repo 文件，立刻开 `codex/*` 分支；
3. 默认**只开分支**，不额外开 worktree；
4. 只有在“并行多需求 / 需要独立运行环境 / 长时间隔离验证”时才额外开 worktree；
5. 开发完成后，至少完成：
   - `npm run check:baseline`
   - 回填受影响核心文档
   - 更新 `AI_HANDOFF_LOG`
   - 清理 `07` 中已解决/失效项
   - 检查 `AI_HANDOFF_LOG` 是否需要归档旧窗口
   - 变更卡结果回填
6. 合回 `main` 后：
   - 删除临时 `codex/*` 分支；
   - 若使用过额外 worktree，移除该 worktree；
   - 本地最终回到干净 `main`。

## 5. 与版本阶段目标的关系
- 每个版本阶段至少有一张 `STG-*` 卡，描述目标、范围、里程碑和退出条件。
- 所有 `MOD/REQ/INV/CFG` 卡必须关联一个 `STG-*`（或说明“独立紧急修复”）。

## 6. 禁止行为
- 禁止先改代码后补变更卡。
- 禁止无 CAP 绑定的业务改动。
- 禁止在变更卡未回填结果时标记“完成”。

## 7. 场景映射（A/B/C/D）
- **A 新模块**：建 `MOD-*` + 对应 `STG-*`；完成后归档到 `archive/changes`，并在 `02` 新增/更新 CAP 卡。
- **B 小需求**：建 `REQ-*`；若不改契约可只回填 `02`，若改接口必须同步 `03`。
- **C 问题排查**：建 `INV-*`；先记录事实与复现，再写修复方案，验收后归档并关联事故/复盘文档。
- **D 配置/数据调整**：建 `CFG-*`；强制记录变更前值、变更命令、回滚值，验收后归档。
