# S03：消息事件重估策略

## 当前结论

这条线不是“再做一个纯新闻推荐器”，而是把消息/公告/问答接到交易决策里，回答两个问题：

```text
候选票为什么涨？
我拿到一条消息后，最快应该看哪些票、怎么跟踪、何时等资金确认？
```

所以它有两条入口：

| 入口 | 业务场景 | 最快可落地形态 |
|---|---|---|
| 量化候选 → 事件解释 | 每日复盘页已经选出票，用户想知道为什么 | 候选票事件解释卡 |
| 消息/公告 → 股票研判 | 用户先看到一条消息，想知道接下来做什么 | 消息触发快速研判卡 |

交易动作仍由量化买卖点和风控决定，LLM 负责解释、分类、归因和研究辅助。

## 已在历史文档中确认过的关键依据

来源：

```text
docs/strategy-rework/REQ-20260425-selection-strategy-rework.md
docs/strategy-rework/_archive/obsolete-root-docs/layer-2-company-event-research.md
docs/changes/REQ-20260412-01-single-stock-news-event-foundation.md
docs/changes/REQ-20260423-01-stock-event-refine-and-selection-fusion.md
docs/changes/MOD-20260424-01-stock-events-current-state.md
```

关键共识：

1. 同样是涨停，纯数字无法解释质量，必须看事件和公司逻辑。
2. 大模型只负责解释和分类，不直接替代量化买卖。
3. 历史回放必须遵守当时可见原则，不能用未来新闻解释过去信号。
4. 公司研究层不能只是新闻摘要，还要解释主营、利润来源、题材、估值弹性、风险点。
5. 突发事件驱动的票可能没有前置吸筹，不能被“必须有吸筹”规则过滤。
6. 事件强但资金未确认时，不直接买；应等待洗盘/分歧后的资金回补确认。
7. 事件层底座已存在：公告 / 问答 / 资讯三类事实源已接通，无 token 也有公共 fallback；当前缺的是事件理解层和选股侧闭环。

## 策略覆盖对象

当前资金流策略容易漏掉这类票：

```text
资金流特征不足
没有标准回调承接
启动前没有明显传统吸筹
涨幅由事件、业绩、政策、题材、公司转型驱动
```

强势样本反推中，已有一批票被标注为“需要消息事件重估策略解释”，例如：

```text
九安医疗
恒铭达
电光科技
节能风电
晶科科技
美利云
江特电机
```

这些不是说一定适合买，而是说明：仅靠当前资金流策略无法解释，需要进入事件重估研究。

## 最快需求 A：候选票事件解释卡

接到现有“每日复盘决策页”：

```text
用户选择日期
→ 页面拿到当日多策略候选
→ 对候选 TopN / 明日可操作 / 观察池读取或触发事件包
→ 生成事件解释卡
→ 展示在右侧复盘页，辅助用户判断是否值得继续研究
```

### 输入

来自当前页面：

```text
symbol
trade_date
strategy_internal_id
candidate_type
量化理由
关键资金/价格指标
```

来自事件层：

```text
stock_events：公告 / 财报 / 监管 / 新闻
stock_event_entities：事件与股票关联
stock_event_daily_rollup：事件聚合
sentiment_events：股吧/评论事件
sentiment_daily_scores：情绪摘要
stock_symbol_aliases：新闻归属辅助
```

必须按时间截断：

```text
published_at <= trade_date 收盘后可见时间
```

如果缺数据，必须明确：

```text
event_context_missing = true
source_unavailable = true/false
```

不能把“源不可用”伪装成“没有新闻”。

## 最快需求 B：消息触发快速研判卡

这是本次补漏重点：用户不是先从候选票进入，而是先“拿到一条消息/公告/董秘回复/新闻”，然后想最快知道下一步该做什么。

### 用户故事

```text
作为盘后复盘用户，
当我看到一条可能影响股价的消息，
我希望系统立刻告诉我：
这条消息对应哪些股票，强度如何，是否可能形成中期重估，
当前资金和价格有没有验证，
明天应该买、观察、等洗盘，还是直接忽略。
```

### 处理链路

```text
输入一条消息或选择一条已采事件
→ 识别关联股票 / 板块 / 题材
→ 判断事件类型、方向、强度、持续性
→ 拉取关联股票最近价格、成交额、L2资金、挂单承接状态
→ 判断“事件与资金是否一致”
→ 输出明日操作建议和跟踪条件
```

### 快速研判输出

| 输出 | 业务含义 |
|---|---|
| 关联股票 | 这条消息真正影响哪些票，避免只看标题误匹配 |
| 事件类型 | 业绩、订单、政策、供需、转型、题材、监管、传闻 |
| 事件强度 | 弱 / 中 / 强 |
| 持续性 | 一日游 / 1~2 周 / 中期逻辑 |
| 资金确认 | L2 主力/超大单是否同步进入 |
| 价格确认 | 是否放量、涨停、突破、回踩承接 |
| 操作节奏 | 可加入观察 / 等洗盘 / 等资金回补 / 明日可操作 / 不参与 |
| 跟踪条件 | 后续哪一天、哪个资金/价格条件出现才升级为买点 |

### 交易规则边界

```text
强事件 + 无资金确认：进入观察，不直接买。
强事件 + 放量启动：不追第一天，观察分歧/洗盘后的承接。
强事件 + 分歧后资金回补：可进入买入确认。
弱事件 + 资金强：按资金策略处理，但提示一日游风险。
事件利空或逻辑证伪：降低持有信心，触发风险提示。
```

## 事件解释卡输出

右侧页面建议增加一个轻量模块，默认折叠，不撑页面。

1. **公司一句话画像**：公司原来做什么，现在市场关注什么，最近有没有业务变化。
2. **近期事件时间线**：公告 / 财报 / 业绩预告 / 问答 / 新闻 / 龙虎榜，只展示当时可见事件。
3. **事件质量判断**：事件类型、强度、持续性、方向、置信度。
4. **估值弹性粗算**：新增利润或市值空间来自哪里，当前涨幅是否透支。
5. **资金与事件一致性**：事件逻辑是否被 L2 资金验证。
6. **对交易的影响**：可继续研究、等洗盘/回补确认、偏一日游、降低持有信心等。

## 结构化字段

| 字段 | 取值 |
|---|---|
| `catalyst_type` | 业绩 / 订单 / 政策 / 行业供需 / 公司转型 / 题材 / 监管 / 传闻 |
| `event_strength` | 弱 / 中 / 强 |
| `time_horizon` | 一日游 / 1~2周 / 中期逻辑 |
| `event_polarity` | 利好 / 利空 / 中性 / 不确定 |
| `confidence` | 0~1 |
| `fund_event_alignment` | confirmed / funds_only / logic_only / conflict / unknown |

`fund_event_alignment` 含义：

| 结论 | 含义 |
|---|---|
| `confirmed` | 事件逻辑强，且 L2 资金同步验证 |
| `funds_only` | 资金强但事件逻辑弱，偏情绪或一日游 |
| `logic_only` | 逻辑强但资金未确认，适合观察 |
| `conflict` | 事件与资金矛盾，谨慎 |
| `unknown` | 事件数据不足 |

## LLM 固定判定模板

大模型必须按固定问题回答：

1. 这家公司主营是什么？最近是否发生业务变化？
2. 当前事件能解释哪些股票或板块？
3. 事件属于短线刺激，还是可能形成中期逻辑？
4. 如果是中期逻辑，利润或估值弹性来自哪里？
5. 当前 L2 资金行为是否验证这个逻辑？
6. 最大风险是什么？
7. 对交易状态的建议是什么：可进场、等洗盘、只观察、不参与？

## 事件驱动启动的特殊路径

这条路径需要独立于传统吸筹：

```text
突发事件
→ 当日/次日涨停或放量启动
→ 后续分歧/砸盘/洗盘
→ 资金回补
→ 二次启动
```

策略重点：

```text
事件是否足够强
事件是否能支撑中期逻辑
启动后资金是否真实承接
第一次分歧后主力是否回补
是否从一日游演变成中期资金接力
```

## 与现有策略的关系

| 策略 | 消息事件重估的作用 |
|---|---|
| 资金流回调稳健 | 给事件质量加权，帮助判断能不能更有信心持有 |
| 趋势中继高质量回踩 | 判断趋势中继背后是否有中期逻辑，而不是纯资金接力 |
| 每日复盘决策页 | 作为解释层进入右侧详情，不直接替代左侧量化候选 |
| 事件层底座 | 提供公告 / 问答 / 新闻事实流、覆盖状态、按需触发能力 |

## 最小验收标准

1. 在每日复盘页点任意候选，能看到事件覆盖状态。
2. 如果事件源缺失，要显示是“源不可用”还是“真没事件”。
3. 对至少一只候选能生成结构化事件解释卡。
4. 输入或选择一条事件后，能生成消息触发快速研判卡。
5. 历史日期只使用当时可见事件。
6. 输出必须包含：事件类型、强度、持续性、资金一致性、风险点、操作节奏建议。
