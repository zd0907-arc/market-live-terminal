# REQ-20260425 选股策略重构需求

## 1. 背景结论

当前选股页已有三类历史逻辑：

- 吸筹前置：`stealth`
- 启动确认：`breakout`
- 出货风险：`distribution`

但它们本质是三个静态线性打分，不能表达“主力资金从吸筹、洗盘、启动、持有、派发到退出”的连续过程。利通电子 `603629.SH` 的案例显示，旧策略在关键阶段没有发出有效确认信号，也没有解决“拿得住”的问题。

## 2. 本轮目标

把选股能力重构为：

```text
盘后 L2 资金生命周期识别 + 中期波段候选 + 持有/退出决策
```

目标问题：

1. 怎么找到利通电子式强势波段股。
2. 找到后怎么判断进场。
3. 进场后怎么每天确认还能拿。
4. 什么时候识别出货/承接失效并退出。

目标持有周期：约 1 周到 1-2 个月。

## 3. 核心产品判断

### 3.1 启动确认不应和吸筹割裂

“启动确认”不是独立策略入口，而是资金生命周期中的阶段跃迁：

```text
观察池 → 吸筹/承接 → 洗盘确认 → 启动确认 → 持有确认 → 风险减仓/退出
```

后续页面可以仍显示“启动确认 Top10”，但底层不应再用一个孤立 `breakout_score` 做榜单。

### 3.2 出货不是吸筹的反向

出货应独立建模，识别的是：

- 高位主动卖压
- 大单/超大单派发
- 买盘承接失效
- 挂单诱多、撤买、加卖
- 放量滞涨与尾盘修复失败

不能简单理解为“净流入为负”或“吸筹分下降”。

### 3.3 数据优势来自盘后 L2

本系统相对普通看盘软件的优势在于：

- 有逐日/5分钟 L2 成交结构。
- 有主力/超大资金分层。
- 有挂单、撤单、OIB、CVD、买盘支撑、卖压等结构化数据。
- 可以盘后批量回测，不依赖主观盯盘。

详见：`data-map-current.md`

重点可用表：

## 5. 新策略方向草案

### 5.1 生命周期状态

建议先定义每日状态，而不是只定义分数：

| 状态 | 含义 | 主要用途 |
| --- | --- | --- |
| `watch` | 有资金异动但趋势未确认 | 加入观察池 |
| `accumulating` | 主力持续承接/吸筹 | 吸筹候选 |
| `shakeout` | 分歧/洗盘但承接未破 | 避免误卖 |
| `launching` | 平台突破/主升启动 | 进场/加仓确认 |
| `holding` | 趋势和资金仍健康 | 拿得住 |
| `distribution_warning` | 高位卖压/承接变弱 | 减仓预警 |
| `exit` | 出货/趋势破坏 | 退出 |

### 5.2 候选发现

候选发现不再只看 `stealth_score`，应综合：

- L2主力/超大净额持续性：`l2_main_net_amount`, `l2_super_net_amount`
- L2主动买强度：`l2_buy_ratio - l2_sell_ratio`
- 订单流不平衡：`oib_delta_amount`, `positive_oib_bar_count`
- CVD：`cvd_delta_amount`, `positive_cvd_bar_count`
- 承接质量：`buy_support_ratio`, `sell_pressure_ratio`
- 价格结构：20/60日区间位置、平台突破、回撤修复
- 成交额/量能：成交额相对20日均值，避免小票噪声

### 5.3 启动确认

启动确认应是：

```text
资金已持续参与 + 洗盘/平台消化完成 + 放量突破/强修复
```

候选因子：

- `close > prev20_high` 或突破平台区间。
- `total_amount / amount_ma20` 放大。
- `l2_main_net_amount` 或 `l2_super_net_amount` 显著转强。
- `positive_l2_net_bar_count > negative_l2_net_bar_count`。
- `oib_delta_amount` 或 `cvd_delta_amount` 同步转正。
- 最近 3 日资金回补覆盖前一轮流出。

### 5.4 持有确认

持有确认每天盘后更新，核心不是重新选股，而是判断是否继续拿：

- 趋势：未跌破关键均线/平台。
- 资金：3/5日 L2 主力净额未连续恶化。
- 承接：`buy_support_ratio` 不弱，`sell_pressure_ratio` 未持续升高。
- 洗盘容忍：急跌后 1-3 日内资金回补、价格修复，则继续持有。

### 5.5 出货/退出

出货模型独立于吸筹模型：

- 高位涨幅已大，但价格推进效率下降。
- `l2_main_net_amount`、`l2_super_net_amount` 多日转负。
- `add_sell_amount` 放大，`cancel_buy_amount` 放大。
- `sell_pressure_ratio` 上升，`buy_support_ratio` 下降。
- `cvd_delta_amount`、`oib_delta_amount` 转负。
- 放量滞涨、冲高回落、尾盘净流入修复失败。

## 6. 回测重构要求

现有回测只有固定持有期，后续必须新增：

1. 动态持有回测：按每日状态决定继续持有/减仓/退出。
2. 信号去重：同一票连续信号只算一次交易周期。
3. 入场口径：信号日后下一交易日开盘/收盘两种口径。
4. 退出口径：出货状态、趋势破位、固定最大持有天数、止损止盈。
5. 评估指标：胜率、平均收益、中位收益、最大回撤、最大浮盈回吐、持有天数、错过主升段比例。
6. 案例回放：必须支持单票逐日状态解释，先以 `sh603629` 验证。

## 7. 第一阶段交付建议

先不急着改前端，第一阶段做策略实验底座：

1. 新增策略版本：`selection_strategy_v2_lifecycle`。
2. 新增离线实验脚本，只读正式库，输出临时实验表或 CSV。
3. 先跑 `sh603629` 单票回放，确认状态机能解释这轮行情。
4. 再跑全市场 `2025-01-02 ~ 2026-04-24` 回测。
5. 结果稳定后再接回选股页。

## 8. 验收标准草案

- 能解释利通电子 2026-01 到 2026-04 的核心阶段。
- 能在主升前或主升初期给出可执行信号。
- 不因为普通洗盘过早退出。
- 高位风险至少能明显早于机械固定持有退出给出预警。
- 回测输出可复现，且每个信号能追溯到字段和公式。

## 9. 本轮新增共识：策略必须可量化验证

用户明确要求：生命周期模型不能停留在概念层，后续每一个状态都必须有可调参数，并能进入回测系统验证。

### 9.1 参数化要求

每类行为都应拆成可配置阈值，例如：

- 什么算 L2 主力资金强：`l2_main_net_amount / total_amount`、`l2_super_net_amount / total_amount` 的阈值。
- 什么算主动买强：`l2_buy_ratio - l2_sell_ratio` 的阈值。
- 什么算挂单承接强：`buy_support_ratio`、`sell_pressure_ratio`、`oib_delta_amount`、`cvd_delta_amount` 的阈值。
- 什么算洗盘：单日跌幅、缩量比例、后续 1-3 日资金回补比例。
- 什么算启动：突破幅度、成交额放大倍数、L2净额强度、主动买入强度。
- 什么算出货：高位滞涨、超大单流出、卖压增强、撤买/加卖、尾盘修复失败。

参数不能写死在业务逻辑里，后续要支持版本化配置和批量回测。

### 9.2 回测产出要求

策略框架搭好后，必须能输出用户可直观看懂的量化结果：

- 成功率 / 胜率
- 平均收益、收益中位数
- 最大回撤、平均回撤
- 最大浮盈、浮盈回吐
- 平均持有天数
- 信号后 5/10/20/40/60 日收益
- 动态退出收益 vs 固定持有收益
- 错过主升段比例
- 过早卖出案例
- 一日游/失败启动案例

回测不仅服务策略调参，也服务产品验证：用户需要看到“如果按这套策略做，历史上大概是什么胜率和预期收益”。

## 10. 本轮新增共识：数字信号必须结合新闻/事件

用户明确指出：同样是涨停，纯数字无法解释其质量。涨停背后的新闻和事件决定了行情性质：

- 是突发事件导致的无吸筹启动？
- 是政策/产业逻辑驱动？
- 是公告、订单、业绩、并购、监管、龙虎榜游资驱动？
- 是短线情绪套利，还是可能演变成中期主线？

### 10.1 新闻/事件在策略中的位置

建议流程不是让大模型直接选股，而是：

```text
数字/L2模型先筛出候选 → 拉取新闻/公告/互动/资讯/龙虎榜等上下文 → 大模型按固定判定套路给出事件解释与进场质量判断
```

大模型负责解释和分类，不直接替代量化信号。

### 10.2 事件驱动启动的特殊路径

用户提出一个重要场景：有些票可能不是提前吸筹后启动，而是突发新闻/事件导致资金突然进场。

这类票不能用“必须先有吸筹”过滤掉。应独立建模：

```text
突发事件 → 当日/次日涨停或放量启动 → 后续洗盘/分歧 → 资金回补 → 二次启动
```

这种路径下，策略重点不是向前找传统吸筹，而是判断：

- 事件是否足够强。
- 事件是否能支撑中期逻辑。
- 启动后资金是否真实承接。
- 第一次分歧/砸盘后，主力是否回补。
- 是否从一日游演变成中期资金接力。

### 10.3 新闻 + 数字综合判断输出

候选票最终应给出结构化判断：

- 技术/L2状态：启动、承接、洗盘、持有、出货。
- 新闻事件类型：公告、业绩、订单、政策、行业、龙虎榜、市场传闻等。
- 事件强度：弱/中/强。
- 持续性：一日游、短线题材、中期逻辑。
- 资金一致性：新闻逻辑是否被 L2 资金验证。
- 操作建议：观察、可进场、等洗盘、继续持有、减仓、退出。

## 11. 当前被用户认可并需保留的策略方向

截至本轮对话，以下方向视为短期记忆，后续设计不得丢失：

1. 用户目标是跟随中小盘资金驱动的中期波段，不是日内交易，也不是长期价值投资。
2. 目标市值范围优先考虑约 50亿-500亿。
3. 目标持有周期约 1周到1-2个月。
4. 核心优势来自盘后 L2：成交、主力/超大资金、挂单、撤单、OIB/CVD、承接/卖压。
5. 策略应识别资金生命周期，而不是三个孤立分数。
6. 启动确认不应与吸筹割裂，但也不能强制要求所有启动前都有传统吸筹。
7. 出货不是吸筹反向，必须独立识别高位派发、卖压、承接失效。
8. 洗盘是核心状态：不能把所有下跌/流出都当成卖出；要看后续资金回补。
9. 利通电子是第一优先案例，用于验证“找得到、拿得住、出得掉”。
10. 后续所有策略结论必须能回测，并以胜率、收益、回撤、持有天数等量化结果验证。
11. 新闻/事件线必须进入综合判断，大模型可用于按固定套路解释候选票质量。

## 12. 本轮新增共识：最终产品形态是历史日完整模拟交易链路

用户进一步明确：未来使用方式不是只看某一天 Top10，而是可以任选一个历史交易日，系统从该日出发完整模拟“选股 → 事件理解 → 进场 → 每日持有判断 → 出场 → 收益统计”。

### 12.1 单日模拟链路

示例：用户选择 `2026-03-02`。

系统应执行：

```text
选择历史日 2026-03-02
→ L2/价格/量能模型筛出候选 TopN
→ 对候选逐个拉取截至 2026-03-02 当时可见的新闻/公告/事件
→ 大模型按固定套路判断事件质量和资金一致性
→ 给出可进场 / 不进场 / 等洗盘等结论
→ 对可进场票生成模拟入场点，例如 2026-03-03
→ 从入场日起逐日模拟盘后持有判断
→ 遇到出货/承接失效/趋势破坏/策略止盈止损等条件，生成模拟出场点，例如 2026-03-09
→ 输出该笔交易收益、最大浮盈、最大回撤、持有天数、出场原因
```

用户希望在系统中直接看到类似：

```text
2026-03-02 信号
2026-03-03 入场
2026-03-09 出场
收益率 X%
出场原因：出货风险 / 承接失效 / 趋势破位 / 止盈止损
```

### 12.2 批量月度回测链路

单日链路跑通后，用户希望按月份批量跑：

```text
对 3 月每个交易日都执行一次完整链路
→ 每天产出可买候选
→ 每只候选模拟入场和动态出场
→ 汇总整个 3 月策略表现
```

月度结果至少包括：

- 触发候选天数
- 总候选数
- 大模型通过数
- 实际模拟交易数
- 胜率
- 平均收益率
- 收益中位数
- 最大单笔收益
- 最大单笔亏损
- 平均持有天数
- 最大回撤
- 资金曲线/累计收益
- 失败案例列表
- 过早卖出案例列表
- 一日游误判案例列表

### 12.3 量化与大模型的职责边界

本轮进一步确认：

- 大模型用于“看懂新闻/事件/逻辑质量”。
- 进场/出场仍必须由量化数据支撑。
- 即使大模型判断新闻驱动较强，也不能脱离 L2 资金和价格结构直接买。
- 每日出场判断主要依赖量化状态：洗盘、承接、出货、趋势破坏、止盈止损。

推荐决策结构：

```text
量化信号给候选和交易状态
新闻/大模型给事件质量和置信度
最终交易动作 = 量化状态 × 事件质量 × 风控参数
```

### 12.4 历史新闻的时间点约束

做历史日模拟时，新闻/公告/事件必须遵守“当时可见”原则：

- 选择 `2026-03-02` 时，只能使用 `published_at <= 2026-03-02 收盘后可见时间` 的信息。
- 不能使用未来新闻解释过去信号，避免未来函数。
- 如果事件数据缺失，应标记为 `event_context_missing`，不能假装已验证。

### 12.5 实盘使用预期

用户最终希望通过历史回测把策略胜率做到至少可接受水平，例如大于 50% 或更高，再考虑实盘使用。

因此策略上线前必须先支持：

1. 单日历史回放。
2. 月度批量回测。
3. 参数版本对比。
4. 数字信号 + 新闻判断的可解释复盘。

## 13. 本轮新增共识：第二层应升级为“公司研究 + 事件解释层”

用户进一步明确：第二层不能只做新闻摘要。量化模型筛出候选后，系统应像研究一家公司一样，对候选票做结构化解释，帮助用户在晚间复盘时形成第二天是否买入/持有的主观决策。

### 13.1 利通电子案例给出的产品启发

用户能在利通电子波动中拿得住，核心不是只看技术指标，而是理解了公司逻辑：

- 原主营是家用消费电子/电视背板等低利润业务，历史盈利能力弱。
- 后续转型算力租赁，这是业务性质改变的大事件。
- 算力规模从约 2.9 万匹变成约 3.3 万匹，租金和利润弹性可粗算。
- 如果算力租赁全部出租，单季利润可能显著提升，进而推导年化扣非利润和估值空间。
- 季报/年报验证利润增长后，市场会重新定价。

这说明：真正能让用户拿住中期波段的，不只是“资金在买”，还包括“用户理解这家公司为什么值得被资金持续炒”。

### 13.2 第二层职责升级

原第二层：

```text
新闻/事件解释
```

升级为：

```text
公司研究 + 新闻/事件解释 + 题材逻辑 + 估值弹性说明
```

这层不一定直接决定买卖，但必须给用户清晰展示：

- 这家公司是做什么的。
- 主营业务和利润来源是什么。
- 最近是否发生业务转型、订单、业绩、政策、行业催化。
- 它属于哪些板块/题材。
- 股价通常受什么因素影响。
- 当前新闻是否能解释资金行为。
- 如果逻辑成立，大致估值空间在哪里。
- 逻辑最大的风险点是什么。

### 13.3 公司研究层建议输出

候选票详情页应输出结构化卡片：

1. **公司一句话画像**
   - 主营业务、核心变化、当前市场关注点。

2. **业务与利润来源**
   - 主营收入来源。
   - 新业务/转型业务。
   - 利润弹性来自哪里。
   - 是否有可量化的产能/订单/出租率/价格/毛利率线索。

3. **近期事件时间线**
   - 公告、财报、互动问答、董秘回复、投资者关系、媒体报道、龙虎榜等。
   - 严格按历史时间点展示，避免未来函数。

4. **题材与板块映射**
   - 算力、AI、消费电子、化工、军工、资源、机器人等。
   - 题材强度与市场热度。
   - 是否处于市场主线。

5. **估值弹性粗算**
   - 不是精确估值，而是帮助用户理解“为什么市场可能重估”。
   - 例如：新增利润 × 合理 PE → 目标市值区间。
   - 必须标记假设来源和不确定性。

6. **资金行为与基本面是否一致**
   - L2资金进入是否能被新闻/业务逻辑解释。
   - 如果资金强但逻辑弱，提示一日游/纯情绪风险。
   - 如果逻辑强但资金不跟，提示等待确认。

7. **用户决策辅助结论**
   - 可关注 / 可进场 / 等洗盘 / 谨慎参与 / 不参与。
   - 主要理由。
   - 关键跟踪点。
   - 风险点。

### 13.4 对交易状态层的影响

公司研究层不会替代量化出入场，但会影响置信度和持有耐心：

- 有中期逻辑 + 资金持续：允许更长持有、更高回撤容忍。
- 纯事件一日游 + 资金不持续：更短持有、更快退出。
- 突发强事件 + 无前置吸筹：允许走“事件驱动启动 → 分歧洗盘 → 回补确认”路径。
- 基本面逻辑被财报/公告验证：提高持有确认权重。
- 逻辑被证伪或事件热度衰退：降低持有置信度。

## 14. 下一阶段细化方向

框架已确认。后续进入细节定义，建议按以下顺序展开：

1. 定义候选发现层：哪些 L2/价格/量能条件能把票筛出来。
2. 定义公司研究 + 事件解释层：需要哪些数据、输出哪些卡片、LLM 按什么套路判断。
3. 定义交易状态层：进场、持有、洗盘、减仓、退出的状态和参数。
4. 定义单日历史回放：输入某日，输出候选、解释、模拟交易结果。
5. 定义批量回测：按月/区间跑完整链路并输出统计。
