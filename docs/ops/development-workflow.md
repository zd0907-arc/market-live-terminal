# 标准开发流程（需求开启 → 合并收口）

> 本文档是当前项目默认开发流程。  
> 目标不是解释所有背景，而是给出**每次需求都照着走的固定动作**。

## 1. 什么时候开始一个新需求
满足任一条件，就应视为一个新需求：
- 要新增/修改 repo 文件；
- 要改变长期事实、契约、能力边界；
- 要修一个会影响用户使用或数据链路的问题；
- 要做一次明确的调查/验证并输出结论。

## 2. 开始前先做什么
1. 先在 `main` 只读排查范围，不直接改文件；
2. 新建变更卡到 `docs/changes/`；
3. 在 `02_BUSINESS_DOMAIN.md` 找到对应 CAP；
4. 判断这次需求属于：
   - `REQ` 小需求
   - `MOD` 模块/能力调整
   - `INV` 问题排查
   - `CFG` 配置/数据调整
   - `STG` 阶段目标

## 3. 分支 / worktree 决策
### 默认规则
- **默认只开一个 `codex/*` 分支**
- **默认不开额外 worktree**

### 只有这三种情况才开 worktree
1. 并行做两条独立需求线；
2. 需要独立运行环境；
3. 需要长时间隔离验证，不能污染当前主工作目录。

## 4. 开发中必须保持什么
1. 代码改动只在当前 `codex/*` 分支进行；
2. 当前需求的过程只写到对应 `docs/changes/*`；
3. 长期事实变化要记下来，准备回流到核心长记忆；
4. 不把聊天记录当长期真相。

## 5. 完成实现后先做什么
至少完成：
1. `npm run check:baseline`
2. 变更卡结果回填
3. 回流受影响的长期事实到核心文档
4. 追加 `AI_HANDOFF_LOG`
5. 清理 `07_PENDING_TODO` 中已解决/失效项
6. 检查 `AI_HANDOFF_LOG` 是否超出当前窗口，必要时归档旧窗口

## 6. 合并前 gate
合并回 `main` 前，至少确认：
1. 当前工作区干净；
2. `baseline` 通过；
3. 版本号一致（若本轮涉及版本变化）；
4. 变更卡已回填；
5. 核心文档没有继续堆过程细节；
6. 当前需求没有遗留“口头待办”。

## 7. 合并后必须做什么
1. 合并回 `main`
2. 删除临时 `codex/*` 分支
3. 如果开过额外 worktree，移除该 worktree
4. 本地最终回到干净 `main`
5. 确认 `AI_HANDOFF_LOG / 07 / README / AI_QUICK_START` 已是最新状态

## 8. 什么时候归档
- 需求完成且结果已回填后，归档 `docs/changes/*`
- 旧 handoff / 旧 pending 超出主窗口后，整理成 archive summary
- 归档不是补丁动作，而是标准收尾动作的一部分

## 9. 当前最容易犯错的点
1. 代码改完了，但长期事实没回流；
2. 已解决的 pending 还留在 `07`；
3. 旧窗口 handoff 不归档，核心日志越堆越长；
4. 合并后分支 / worktree 没收口；
5. 把过程卡当长期真相。

## 10. 一句话标准
**每个需求都必须做到：有卡开始、在分支实现、用 baseline 验证、把长期事实回流、把短记忆清理、最后回到干净 main。**
