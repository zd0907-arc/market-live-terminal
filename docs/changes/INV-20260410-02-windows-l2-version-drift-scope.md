## 4. 关键结论

### 4.1 Windows 正式跑数链路落后不是“小版本”，而是“阶段性落后”
- 目前 Windows 跑数机上的 `l2_daily_backfill.py` 仍是**早期 Phase 2 版本**：
  - 只做成交级 `L1/L2 主力/超大单`
  - 不解析真实挂单事件层
  - 没有 `build_standardized_order_events(...)`
  - 没有 `total_volume + add/cancel + l2_cvd_delta + l2_oib_delta` 的完整写库逻辑
- Windows 跑数机上的 `l2_history_db.py` 也停在**仅补 `quality_info`** 的阶段：
  - 有 `quality_info`
  - **没有** `total_volume / l2_add_* / l2_cancel_* / l2_cvd_delta / l2_oib_delta`

### 4.2 它大致对应仓库历史上的什么阶段
- 对照文档时间线：
  1. `2026-03-14 ~ 03-15`：完成基础 `5m + daily` 正式回补
  2. `2026-03-15`：补 `quality_info`
  3. `2026-03-19`：才把 `total_volume + add/cancel + l2_cvd_delta + l2_oib_delta` 纳入正式 schema / API
- Windows 当前代码状态更接近：
  - **介于 `2026-03-15` 与 `2026-03-19` 之前**
- 结论：
  - Windows 端不是“只差一个 bugfix”
  - 而是**整段没有升级到事件因子时代**

### 4.3 现有正式库的影响范围
- Windows 正式库当前覆盖：
  - `2026-03-02 ~ 2026-03-13`
  - `history_daily_l2 = 76,749`
  - `history_5m_l2 = 3,461,334`
  - `8,111` symbols
- 因为 Windows schema 本身没有事件列，所以这批正式结果：
  - **全体都缺** `total_volume`
  - **全体都缺** `l2_add_* / l2_cancel_* / l2_cvd_delta / l2_oib_delta`
- 这意味着：
  - 对这 `10` 个交易日而言，问题不是“少数票异常”
  - 而是**整批正式入库结果都不具备事件层能力**

### 4.4 但不是所有历史都需要重跑
- `2025-01 ~ 2026-02`
  - 原始源本身只有成交级
  - 不存在真实挂单事件层
  - 所以这段**不需要为 add/cancel 而重跑**
- 真正值得重跑的，是：
  - **`2026-03+` 这段有真实挂单 raw 的窗口**

## 5. 原始包格式问题范围

### 5.1 不是全市场统一单一编码
- `sz000833 / 2026-03-11 / 逐笔委托.csv`
  - `委托类型` 抽样结果：`0 / 1 / U`
- `sh600519 / 2026-03-11 / 逐笔委托.csv`
  - `委托类型` 抽样结果：`A / D / S`
- `sh603629 / 2026-03-11 ~ 03-18 / 逐笔委托.csv`
  - `委托类型` 抽样结果：`A / D / S`
- `sh603629 / 2026-04-01 / 逐笔委托.csv`
  - `委托类型` 抽样结果：`A / D / S`

### 5.2 含义
- 旧文档与旧实现中，事件解释冻结为：
  - `0 = add`
  - `1/U = cancel-like`
- 但实际 raw 至少存在第二类编码体系：
  - `A = add`
  - `D = cancel`
  - `S = 特殊状态行（应过滤）`

### 5.3 结论
- “事件码 bug”不是利通个股偶发，而是**供应商 raw 存在多套编码体系**
- 因此未来正式修复必须采用：
  - **兼容多映射表**
  - 不能再把 `0/1/U` 写死成唯一世界
- 当前样本层面可形成一个**待验证假设**：
  - 深市样本更像 `0/1/U`
  - 沪市样本更像 `A/D/S`
  - 但这还不是最终规则，仍需事件码范围审计脚本继续扩大样本验证

## 6. 对工程量的真实判断

### 方案 A：最小修复
- 目标：先保证利通与少量研究样本可复盘
- 范围：
  - 只补单票、只补关键日
- 优点：
  - 快，马上能做深复盘
- 缺点：
  - 不能解决全市场选股研究的底层一致性

### 方案 B：中等规模修复（推荐）
- 目标：把**最近有真实挂单 raw 的窗口**修到可研究
- 范围建议：
  - `2026-03-02 ~ 2026-04-10`
  - 当前 raw 已见 `29` 个交易日
- 动作：
  1. 升级 Windows 代码与 schema
  2. 先对最近 `29` 个交易日重跑正式 L2
- 优点：
  - 能支撑利通复盘
  - 也能支撑后续选股研究 / 相似失败样本筛选
- 缺点：
  - 是一项正式工程，不是当天就能全做完

### 方案 C：大规模全历史重做
- 目标：所有历史口径完全统一
- 结论：
  - **当前没必要**
  - 因为 `2025-01 ~ 2026-02` 本来就没有真实挂单事件 raw，重跑也补不出 add/cancel

## 7. 当前建议

### 7.1 先后顺序
- **不是二选一**
- 推荐并行两条线：
  1. **利通单票深复盘线**：继续用已修关键日先做研究
  2. **Windows 正式链路修复线**：先把范围和升级方案做清楚，再分批重跑 `2026-03+`

### 7.2 为什么不建议“先全修完再复盘”
- 因为：
  - 利通关键日已经能先复盘；
  - 全局修复至少是一个中等工程；
  - 如果等全修完才研究，会把节奏拖太久。

### 7.3 为什么也不能只做利通、不修底层
- 因为：
  - 你后续要做的是选股系统，不是只做一个案例；
  - 当前 Windows 正式链路的事件层缺失，会直接影响后续研究质量。
