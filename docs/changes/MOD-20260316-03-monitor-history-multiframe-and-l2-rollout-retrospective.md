## 5. 本次项目里最重要的正确决策
### 5.1 不在旧版上硬改，而是开新版分叉
这是本次最关键的风险控制动作。

收益：
- 老功能不被拖下水；
- 允许我们在新版上反复试错：先做日线、再收敛成历史多维、再重做三副图；
- 用户线上仍有旧版可回退，不会因为一个未成熟视觉方案卡死整个盯盘页。

如果当时直接在旧版日线/30m 上做侵入式改造，这次中途的多轮返工会非常危险。

### 5.2 提前把产品结构冻结成“当日分时 + 历史多维”
这次如果继续沿用最初“三个新版 Tab 都分别重做”的路径，后面会出现三个问题：
1. 30m 与日线重复设计；
2. 数据查询口径割裂；
3. 后续 5m/1h 再扩会越来越乱。

把结构收敛成：
- 新版一级：`当日分时` / `历史多维`
- 历史多维二级：`5m / 30m / 1h / 日`

这是正确的长期架构，后面复盘页并库也因此有了统一方向。

### 5.3 正式历史只认 `5m + 1d`
这一点在数据层上非常关键。

收益：
- `5m` 成为所有历史多维的统一最细底座；
- `30m/1h` 不再需要单独落库、单独回补、单独校验；
- 盘后回补与前端展示的复杂度被压到可控范围。

### 5.4 preview / finalized 明确分层
这次如果没有把：
- 今日盘中 preview
- 次日正式 finalized

明确拆开，用户会看到最危险的情况：**今天的数据看起来像正式历史，但其实只是 L1 实时口径。**

我们最终冻结的规则是正确的：
- preview 只允许 L1；
- finalized 才允许完整 L2；
- today 未结算时 tooltip 必须明确提示“当前仅 L1 实时口径，L2 待盘后覆盖”。

### 5.5 接受错误，但不能无感知
最初曾尝试“尽可能修完数据错误”。后面发现这是错误方向。

正确方向是：
- 错误数据可以存在；
- 但必须被识别、暴露、解释；
- 缺失必须占位，不得静默消失；
- 异常不得伪装成 0 或正常 finalized 值。

这直接催生了后来的：
- `quality_info`
- `is_placeholder`
- 前端统一 `!` 提示
- tooltip 显示质量文案

这是本次最成熟的数据治理决策。

## 6. 中途踩过的主要坑
### 6.1 坑一：前端视觉先行，但正式数据还没到位
早期我们先看了日线融合图的效果样稿，并且用局部样本做了预览，这对定方向有帮助；
但当历史多维真正扩展到 `5m / 30m / 1h / 日` 时，真实 `2026-03` 正式数据还没完全入库，于是出现：
- 设计方向对，但页面没数；
- 样稿好看，但默认视窗、tooltip 密度、空态都不准；
- 很难判断问题在前端还是在数据底座。

教训：
- 样稿可以先走；
- 但真正上线统一模块前，至少要有一段可用正式样本覆盖做联调。

### 6.2 坑二：Windows 本机父进程分片并发不稳定
`backend/scripts/l2_day_sharded_backfill.py` 这条路在首轮试跑中出现：
- `partial_done`
- `symbol_count=0`
- `rows_daily=0`

结论非常明确：
- 不是单个 symbol 数据异常；
- 是编排层不稳定；
- 不适合做正式定时主路径。

后面切换为：
- **Mac / 控制端多 SSH 会话 + 单日 8 worker shard**

这条路才稳定下来。

### 6.3 坑三：以为“43 个失败样本”是简单修复问题
最开始看到 `43` 个失败样本，直觉上像“补几个特例就行”；
但深入后发现这背后其实暴露的是：
- OrderID 对齐策略过于刚性；
- 上交所/不同品种的现实数据并不总能完美双边对齐；
- 如果设计成“单边对不上也整只失败”，正式表会出现大量局部缺口。

最终才把策略调整为：
- **单边 0 overlap 可回退**；
- **双边 0 overlap 才硬失败**。

### 6.4 坑四：`105` 个空结果样本最开始没有被系统化记录
这批样本一开始最大的问题不是“空”，而是“容易被静默漏掉”。

风险在于：
- 它们不像显式失败那样容易被发现；
- 前端表现可能只是“少了一个点”；
- 事后很难知道是停牌、无成交、还是源数据异常。

后面把“无有效 bar”也写入 `l2_daily_ingest_failures`，才真正把问题纳入统一 repair/review queue。

### 6.5 坑五：前端第一版历史多维图层没有真正实现“双柱”
这个问题最终暴露得很直接：
- 你在生产看到“每个点只有一根柱子”；
- 同时 ECharts 原生 legend 很挤，tooltip 也跑出屏幕。

这说明第一版实现虽然“功能上接近”，但没有达到视觉与交互要求。

根因：
- 仍试图靠普通 bar series + `barGap` 微调；
- 图层控制不够稳定；
- tooltip 内容过长、边界控制不足；
- 图表内部 legend 占用太多视觉空间。

最终解决方式不是微调，而是重构：
- 三张副图统一改 `custom series`；
- 去掉原生 legend，改 React 自定义图例；
- tooltip 改为受边界约束的紧凑卡片。

## 7. 数据错误问题的最终认识
### 7.1 43 个显式失败样本是什么意思
这批样本的典型模式是“单边 0 overlap”：
- 买边完全对不上、卖边能对上；或
- 卖边完全对不上、买边能对上。

它不意味着这只股票整天都错，也不意味着整批数据都不能用；
它更像是：
- 某一侧母单号链路缺失；
- 但另一侧仍有足够信息可以聚合。

如果此时一刀切判失败，就会把可用信息也丢掉。

所以最后的策略是正确的：
- 允许单边回退；
- 同时写 `quality_info`；
- 告诉前端“值可能偏小”，而不是静默假装完全准确。

### 7.2 105 个空结果样本是什么意思
这批样本不能简单等同于“数据错了”。它可能包含：
- 停牌；
- 当日确实极少/无有效逐笔；
- 原始文件存在但未形成有效正式 bar；
- 源包异常。

所以它们不应该统一按“失败修复”处理，而应拆成：
- **review queue**：继续区分真实无交易 vs 数据异常；
- **查询层 placeholder**：让前端知道这里该有点，但当前缺失；
- **quality_info**：把缺失原因用人类可读文案暴露出来。

### 7.3 为什么最后没有继续追求“全修完”
因为投入产出比不高，而且会带来两个更大的风险：
1. 为少数不重要样本把全系统做复杂；
2. 为追求“表面完美”而引入更多隐藏逻辑，反而更难维护。

所以最终策略是：
- 对会污染口径的错误必须修；
- 对不影响整体可用性的局部缺失，优先做“可见化 + 可解释化”。

这个决策非常务实，也符合真实交易系统的风格。

## 8. 前端历史多维为什么最后能成功
### 8.1 成功的不是“第一版实现”，而是“最后一版重构”
最终生产上可用的关键，不在于最早做出了三张副图，而在于最后承认第一版不够好，然后做了这几件事：
1. 三张副图统一成同一视觉语言；
2. 恢复“左超大 / 右主力”真双柱；
3. 深色 L2 底柱、浅色 L1 芯柱；
4. tooltip 卡片分层展示；
5. tooltip 受边界约束，不再出框；
6. 版本号每次发版可见，避免线上验证混淆。

### 8.2 为什么 custom series 是对的
普通 bar series 在这种需求下有三个天然问题：
- 左右分组不稳定；
- L2/L1 重叠关系不好精确控制；
- 三张副图很难保持完全同构。

改成 custom series 后：
- 每个点的左右位置由我们自己画；
- L2/L1 的包裹关系可控；
- 主图、净流入、买卖力度三层都能复用同一渲染思路。

这也是为什么最后效果终于和你最早确认的样稿一致。

## 9. 发布过程中的教训
### 9.1 版本号必须每次同步更新
你多次指出“生产版本号没变化”，这是完全正确的。

这次之后可以明确成硬规则：
- 只要发生产，就必须同步更新：
  - `package.json`
  - `src/version.ts`
  - `README.md`
- 并且前端页面上要能直接看到版本号。

否则会出现最糟糕的情况：
- 代码可能变了；
- 用户却无法判断自己看到的是不是最新版。

### 9.2 生产问题不能只凭“我觉得应该修好了”
这次“只有一根柱子”的问题就是典型例子。

虽然之前已经提交过一次“修复双柱”的代码，但生产上你看到的仍然是一根柱子。
这说明：
- 提交成功 ≠ 效果正确；
- 接口有数据 ≠ 前端渲染对了；
- 必须回到样稿和真实页面逐项核对。

### 9.3 本地验证后再发生产是对的
你要求“先本地改好、本地验过再发生产”，这一点直接避免了更多线上反复试错。

以后类似“高视觉要求 + 交互要求强”的变更，都应默认走：
- 本地样稿校正
- 本地真实数据验证
- 再发生产
