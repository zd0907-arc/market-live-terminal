# REQ-20260314-02-daily-postclose-l2-fusion

## 2. 背景与目标
- 背景：现有系统已持有 `2025-01-01 ~ 2026-02-28` 的离线 L2 历史包回放结果；你计划新增购买“每日盘后更新”的 L2 日包，并要求把它与现有盯盘页、复盘页、以及未来每日沉淀数据统一融合。
- 本轮先做两件事：
  1. 先验真 Windows 上已落地的 `2026-03-11` 单日日包格式是否可用；
  2. 基于验真结果，形成下一阶段独立开发分支的融合方案与切入顺序。

## 3. 方案与边界
- 做什么：
  - 验证 `D:\\MarketData\\20260311\\20260311\\{symbol}.(SZ|SH|BJ)` 目录结构；
  - 抽样校验 `行情.csv / 逐笔成交.csv / 逐笔委托.csv` 的编码、列名、单位、OrderID 完整性；
  - 输出与当前 ETL/DB/页面的兼容性差异；
  - 提出“生产盯盘 + 深度复盘 + 云端存储边界”的分阶段融合方案；
  - 新增验证脚本 `backend/scripts/inspect_daily_l2_package.py`，作为后续每日新包准入检查入口。
- 不做什么：
  - 本轮不直接把新日包写入生产 `market_data.db`；
  - 本轮不把 Tick/L2 原始逐笔全量同步上云；
  - 本轮不改动现有线上盯盘页/复盘页展示逻辑。

## 4. 执行步骤（按顺序）
1. 远程检查 Windows `D:\\MarketData\\20260311` 结构，确认目录层级与样本标的文件。
2. 抽样 `000833.SZ`，验证三类 CSV 的字段、编码、单位、OrderID 对齐关系。
3. 对照当前 `sandbox_review_etl.py` / `market_data` 契约，列出不兼容点。
4. 形成下一阶段融合设计：本地存储分层、云端派生表、盯盘/复盘页面接入路径、实施顺序。

## 5. 验收标准（Given/When/Then，绝对时间）
- Given `2026-03-14 16:05`，When 远程检查 `D:\\MarketData\\20260311`，Then 确认日包结构为 `20260311/20260311/{symbol}`，且单标的含 `行情.csv`、`逐笔成交.csv`、`逐笔委托.csv` 三类文件。
- Given `2026-03-14 16:08`，When 抽样 `000833.SZ`，Then 验证 `逐笔成交.csv` 具备 `叫买序号/叫卖序号` 且可与 `逐笔委托.csv::交易所委托号` 完整对齐，说明可做 L2 母单级聚合。
- Given `2026-03-14 16:12`，When 用当前 `sandbox_review_etl.py` 假设直接读取该文件，Then 会因“日期提取失败 + 中文列名/字段别名未映射”而不兼容，说明必须单独加新格式适配层。

## 6. 风险与回滚
- 风险：
  1. 新日包与旧历史包并非同一目录/列名规范，若强行复用旧 ETL，容易把日期、价格单位、OrderID 口径做错；
  2. Tick 原始逐笔不能上云，意味着生产链路必须做“本地原始层 + 云端派生层”双层设计；
  3. 若盘后日包未来缺失 `逐笔委托.csv` 或 OrderID 断裂，则无法保留真实 L2 母单口径，只能降级到 L1。
- 回滚：
  - 本轮仅新增验证脚本与文档，不改生产表结构；若后续方案调整，直接废弃本变更卡并重新开新卡即可。

## 7. 结果回填
- 实际改动：
  - 新增日包验证脚本：`backend/scripts/inspect_daily_l2_package.py`
  - 新增本变更卡：`docs/changes/REQ-20260314-02-daily-postclose-l2-fusion.md`
- 验证结果：
  - Windows 已确认 `2026-03-11` 日包存在，外层路径为 `D:\\MarketData\\20260311`，内层股票目录根为 `D:\\MarketData\\20260311\\20260311`；
  - 样本 `000833.SZ` 文件规模：`行情.csv=4103` 行、`逐笔成交.csv=82925` 行、`逐笔委托.csv=88317` 行；
  - `逐笔成交.csv` 字段为 `成交价格/成交数量/BS标志/叫卖序号/叫买序号`，价格需 `/10000` 还原为元，数量已是股；
  - 过滤非成交记录后，样本有效成交 `64166` 行，买卖 OrderID 覆盖率 `100%`，且均能在 `逐笔委托.csv::交易所委托号` 中找到；
  - 当前 `sandbox_review_etl.py` 直接读取该新格式会失败：既识别不出日期，也识别不出 `成交数量/叫买序号/叫卖序号`。
- 遗留问题：
  1. 需要新增“日包适配层”，把新中文列名映射到统一内部字段；
  2. 需要定义生产态“本地原始层 + 云端派生层”的存储边界；
  3. 需要把盯盘页 30m / 日线累计 / 复盘页查询口径统一成同一套派生事实表。
