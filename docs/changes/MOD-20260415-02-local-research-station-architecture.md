# MOD-20260415-02 本地研究站架构重规划（云端只保留盯盘）

> **当前真实方案入口。**
> 若要了解这轮数据治理完成后系统应该怎么运行，优先看本卡；此前关于“完整 atomic 切生产”的文档，统一视为历史过程记录。

## 1. 背景

截至 `2026-04-15`：

- Windows 正式主板 atomic 全量库已完成，文件约 `38.2GB`；
- 云端系统盘 `50G`，剩余仅约 `7.6G`；
- 即使删除云端旧 `market_data.db + sandbox + bak`，也仍不足以稳定承载完整 atomic 正式库；
- 用户当前真实使用习惯也已经明确：
  - **云端/手机** 主要用于临时看盘；
  - **Mac / Windows** 才是主要工作环境；
  - 复盘与选股属于研究型工作，不必强行上生产。

因此，本轮方案从“把 atomic 全量切到生产”调整为：

> **云端只保留轻量盯盘入口；Windows 做数据主站；Mac 做本地研究工作台。**

---

## 2. 新的三端职责冻结

### 2.1 云端（Production Cloud）
只承担：
- 盯盘页 / 手机临时查看；
- 轻量查询接口；
- 当前正式生产入口与基础回滚锚点。

明确不再承担：
- 全量 atomic 主库；
- 复盘研究主链路；
- 选股研究主链路；
- 大规模历史回测计算。

### 2.2 Windows（Data Factory / Source of Truth）
承担：
- 原始包存储；
- 每日盘后正式跑数；
- 全量 atomic 主库；
- 选股研究结果产出；
- 各类研究型中间产物的主落盘。

它是：
> **数据真相源 + 跑数工厂**

### 2.3 Mac（Local Research Workstation）
承担：
- 本地前后端开发运行；
- 复盘页；
- 选股研究页；
- 文档与策略迭代；
- 从 Windows 同步“研究可读快照”。

它是：
> **你真正每天使用的研究工作台**

---

## 3. 数据流重新冻结

### 3.1 主数据流
1. Windows 下载 / 保存 raw 日包；
2. Windows 跑 atomic / selection / 研究派生产物；
3. Windows 生成给 Mac 使用的**裁剪快照**；
4. Mac 拉取快照到本地 `data/`；
5. Mac 本地前后端直接读本地快照库；
6. 云端仅继续保留盯盘所需轻量数据链路。

### 3.2 为什么不让 Mac 直接远程读 Windows sqlite
不推荐，原因：
- sqlite 跨网络直读不稳；
- 容易遇到锁；
- 查询延迟与断连问题会直接破坏页面体验；
- 不适合未来把 Mac 本地页面当成主工作台长期使用。

### 3.3 为什么不把 38G atomic 全量下载到 Mac
不推荐，原因：
- 传输与同步成本太高；
- 占本地磁盘；
- 日常复盘/选股并不需要整库全量常驻；
- 更合理的是 Windows 保留全量，Mac 只拿“可研究、可展示、可回放”的裁剪快照。

---

## 4. Mac 需要同步什么，不需要同步什么

### 4.1 必同步（P0）
1. `selection_research.db`
   - 直接支撑选股页；
   - 体积小；
   - 每天盘后可全量覆盖同步。

2. 复盘所需的**裁剪历史库**
   - 只保留 Mac 页面会直接查询的表 / 字段 / 时间窗；
   - 先覆盖最近窗口与关注票；
   - 不追求一上来就复制 Windows 全量 38G。

3. 必要的股票元数据
   - `stock_universe_meta` 或等价映射；
   - 保证 Mac 侧页面能稳定显示名称、代码、基础信息。

### 4.2 暂不同步（P1 以后再评估）
- Windows 全量 atomic 主库；
- 原始逐笔包；
- 大而全的历史研究中间库；
- 不直接服务 Mac 页面显示的重型明细表。

## 5. 本地研究站的推荐运行形态

### 5.1 云端
- 保持现有生产代码，但定位降级为：**盯盘应急版**；
- 不再把“复盘/选股必须上生产”作为当前目标；
- 新研究型改动优先在本地验证，不以生产发布为第一目标。

### 5.2 Windows
每日盘后流程冻结为：
1. raw 日包准备；
2. atomic 正式跑数；
3. selection 结果生成；
4. 导出 Mac 所需裁剪快照；
5. 提供 Mac 拉取入口（SSH/SCP/rsync over Tailscale）。

### 5.3 Mac
每天使用流程冻结为：
1. 先拉取最新快照；
2. 启动本地后端；
3. 启动本地前端；
4. 复盘/选股都只读本地快照；
5. 若需要更深的全量研究，再回到 Windows 侧做专题计算。

## 6. 这次方案调整带来的直接变化

### 6.1 对生产的变化
- 生产不再是 atomic 全量落地的第一优先级；
- 生产继续可用，但主要承担轻量盯盘；
- 生产版本管理仍要规范，但“新研究功能一做完就上生产”的节奏取消。

### 6.2 对开发节奏的变化
- 以后优先判断：这次改动是“云端盯盘能力”还是“本地研究能力”；
- 复盘/选股/研究页优先在 Mac 本地闭环；
- Windows 与 Mac 之间的数据同步，成为新的关键基础设施。

### 6.3 对数据治理主线的变化
- 数据治理主线不变，仍然是“把 raw 沉淀成稳定 atomic 层”；
- 但消费层目标从“云端全量切换”改为“Windows 真相源 + Mac 本地消费”；
- 后续更关注：
  - 裁剪库怎么设计；
  - 哪些表必须同步；
  - 每天盘后怎样一键完成 Windows -> Mac 更新。

## 7. 新的 P0 / P1 / P2 路线

### P0：冻结架构与运行模式
- 云端只保留盯盘；
- Windows 做数据主站；
- Mac 做研究工作台；
- 停止把“full atomic 上云”作为当前主线目标。

### P1：打通 Windows -> Mac 研究快照同步
- 明确同步目录；
- 明确同步表/库；
- 明确是“全量覆盖”还是“增量替换”；
- 明确 Mac 本地读取路径。

### P2：让 Mac 本地复盘/选股正式切到新数据
- 复盘页改读本地研究快照；
- 选股页改读本地 selection 结果；
- 本地形成真正稳定的研究环境。

### P3：未来再评估云端是否需要扩容
只有在以下条件同时成立时，才重新考虑：
- 你明确希望手机端也能看复盘/选股；
- 本地研究模式已经稳定；
- 云端扩容带来的成本与收益清晰可接受。
