## 2. 背景与目标
- 背景：
  - 当前盘中盯盘依赖腾讯/实时链路；历史日K资金博弈分析主要依赖腾讯/新浪口径；
  - 复盘页 L1/L2 研究能力目前运行在独立 sandbox 数据域，尚未成为生产底座；
  - Windows 已验证新的每日盘后 L2 日包格式可用，但目录组织与现有 ETL/生产 DB 均未收敛。
- 本阶段目标：
  1. 先完成一套可直接实施的文档体系：总方案 + 分阶段需求卡；
  2. 把数据库改造、目录规范、回补模型、查询优先级写到“实施者无需再做设计决策”的程度；
  3. 明确生产终态：**盯盘页 / 日K / 复盘页共用同一套正式 L2 历史底座**；
  4. 明确历史能力增强方向：**同一份 L2 原始行情同时派生 L1 与 L2 指标，供后续页面对比分析**。

## 3. 总体方案与边界
- 做什么：
  - 历史正式层只落 **5m + 日线**，不落 1m；
  - Windows 保留原始日包；云端不存原始 `逐笔成交/逐笔委托/行情`；
  - 当天盘中继续实时链路；次日及以后统一查正式 L2 历史；
  - `15m/30m/1h` 查询统一由 `5m` 聚合，不再为每个周期单独持久化；
  - 复盘页未来迁到生产 L2 底座，sandbox 仅保留实验用途；
  - 建立月/日两级目录规范和“按日覆盖重算”的回补机制。
- 不做什么：
  - 本阶段不直接实现生产代码；
  - 本阶段不定死每个页面未来的 L1/L2 对比交互样式；
  - 本阶段不把原始 Tick/委托上云。

## 4. 数据底座决策（必须写死）
1. **历史最细粒度 = 5m**
   - 生产正式历史层不存 1m；
   - 云端正式持久化只保留 `history_5m_l2` 与 `history_daily_l2`；
   - `15m/30m/1h` 统一查询时由 `history_5m_l2` 聚合。
2. **当日实时 ≠ 次日正式值**
   - 当日盯盘继续看腾讯/实时；
   - 盘后导入的 L2 日包是次日开始的正式历史值。
3. **同源双派生**
   - 同一份盘后 L2 原始行情必须同时产出：
     - L1 指标（按单笔金额阈值）
     - L2 指标（按订单号/母单聚合）
   - 后续所有历史分析模块都从同一底座取数，不再维护多套独立正式口径。
4. **生产与实验分层**
   - 生产正式底座：`market_data.db` 中新增 L2 历史表；
   - sandbox：继续保留，用于实验、验证、特殊研究，但不再作为长期正式页面数据源。

## 5. 数据库与目录设计总览
### 5.1 Windows 原始目录规范
- 统一结构：`D:\MarketData\YYYYMM\YYYYMMDD\{symbol}.SZ\`
- 规则：
  - 月目录：`YYYYMM`
  - 日目录：`YYYYMMDD`
  - 股票目录：`000833.SZ / 600519.SH / 430047.BJ`
  - 固定三文件：`行情.csv / 逐笔成交.csv / 逐笔委托.csv`
- 兼容要求：
  - 对已存在的裸日目录（如 `D:\MarketData\20260311`）视为“过渡态”，应在 Phase 2 脚本中支持迁移/规范化检测；
  - 新下载数据一律按月目录归档。

### 5.2 生产正式表
- `history_5m_l2`
  - 主键：`PRIMARY KEY(symbol, datetime)`
  - 字段下限：
    - `symbol, datetime, source_date`
    - `open/high/low/close`
    - `total_amount`
    - `l1_main_buy/sell`
    - `l1_super_buy/sell`
    - `l2_main_buy/sell`
    - `l2_super_buy/sell`
  - 约束：
    - 不强制持久化 net 字段，查询侧可派生；
    - 必须保证同一 `symbol+datetime` 唯一。
- `history_daily_l2`
  - 主键：`PRIMARY KEY(symbol, date)`
  - 字段下限：
    - `symbol, date`
    - `open/high/low/close`
    - `total_amount`
    - `l1_main_buy/sell/net`
    - `l1_super_buy/sell/net`
    - `l2_main_buy/sell/net`
    - `l2_super_buy/sell/net`
    - `l1_activity_ratio / l1_super_ratio`
    - `l2_activity_ratio / l2_super_ratio`
    - 日K资金博弈分析所需 `buy/sell ratio`
- 回补状态表
  - `l2_daily_ingest_runs`
  - `l2_daily_ingest_failures`
  - 目标：记录整日回补的开始/结束/状态/错误/覆盖行数。

### 5.3 覆盖写契约
- 盘后 L2 是正式值；
- 对同一 `symbol + trade_date`：
  - `history_5m_l2`：整日删后重写；
  - `history_daily_l2`：整日删后重写；
- 禁止 append 叠加，避免重复回补污染。

## 6. 模块改造范围
### 6.1 盯盘页
- 当天：继续实时链路；
- 历史模式：读取 `history_5m_l2`；
- `30m/1h`：由 `history_5m_l2` 聚合。

### 6.2 日K资金博弈分析
- 当天：继续腾讯/实时口径；
- 历史日：改查 `history_daily_l2`；
- 未来增强：在同一历史日结果中支持 **L1 vs L2 对比分析**。

### 6.3 复盘页
- 终态改查生产 `history_5m_l2`；
- sandbox 保留：
  - 算法实验；
  - 口径验证；
  - 特殊研究样本；
- 未来复盘页的 L1/L2 对比直接建立在生产底座之上。

## 7. 分阶段实施清单
1. **Phase 1 — 数据库与目录规范**
   - 先完成 schema、目录命名、回补模型、状态表、覆盖写契约。
2. **Phase 2 — Windows 盘后日包适配与回补链路**
   - 解析新日包、生成 `5m + daily` 的 L1/L2 双派生、支持强回补。
3. **Phase 3 — 生产历史查询切换**
   - 盯盘页历史模式、日K资金博弈分析改查生产正式 L2 历史。
4. **Phase 4 — 复盘页并库**
   - 复盘页从 sandbox 正式迁到生产 L2 底座。

## 8. 风险与回滚
- 风险：
  1. 若目录规范不先统一，后续自动发现/补跑会持续出现日期识别和重复归档问题；
  2. 若继续长期维持 sandbox 与生产双正式库，后续每次口径修复都要双写双修；
  3. 若历史只沉淀 L2 不沉淀 L1，将来页面做对比还要再次回源重算。
- 回滚：
  - 本阶段仅落文档，不改代码，不动生产数据；
  - 如设计调整，只需更新方案卡与阶段卡，不涉及生产回滚。
