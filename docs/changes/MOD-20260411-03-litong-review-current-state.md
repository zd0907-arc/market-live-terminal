## 4. 当前已经完成的事情

## 4.1 已完成原始数据分段核验
已经确认：

### `2025-01 ~ 2026-02`
- raw 是成交级
- 可以做：
  - 主力 / 超大单
  - 5m / 日线
  - 成交节奏
- 不能做：
  - 真实挂单事件层

### `2026-03+`
- raw 同时包含：
  - `逐笔成交`
  - `逐笔委托`
- 所以真实可做：
  - `add/cancel`
  - `cvd`
  - `oib`

---

## 4.2 已完成解析 bug 定位
这次非常关键的一个结论是：

- 事件层缺失，不是因为 raw 没有；
- 而是旧清洗逻辑把事件码识别错了。

已确认至少两套事件码：
- `0 / 1 / U`
- `A / D / S`

对应处理应为：
- `0 -> add`
- `1 -> cancel`
- `U -> cancel`
- `A -> add`
- `D -> cancel`
- `S -> ignore`

## 5. 这条线查出来的真实问题

## 5.1 利通问题不是单票偶发 bug
利通身上暴露的问题，说明的是整个数据链路问题：

1. 原始事件码不统一；
2. Windows 正式链路落后；
3. 本地 / Windows / 云端数据漂移；
4. 原始 `OrderID` 存在部分缺失。

也就是说：
- 利通是一个样本；
- 但它暴露的是**全局治理问题**。

---

## 5.2 利通最近窗口虽然补齐，但不能误认为“全局修好了”
现在只是：
- **利通单票本地窗口补齐**

不是：
- 全市场 `2026-03+` 已经都补齐
- Windows 正式库已经全部升级完成

所以不能把利通当前状态误判成“整套系统已完成数据治理”。

---

## 5.3 高位阶段要带质量标识看
利通 `2026-03-02 ~ 2026-04-10` 很多日子带：
- `quality_info = OrderID 部分缺失，L2 数值可能偏小`

这意味着：
- 趋势、强弱、结构能看；
- 但不要把绝对值当完美法证。

## 7. 当前第一版结论

### 7.1 为什么它能走成
第一版理解不是：
- 简单持续净流入

而更像：
1. 启动前 L2 已有埋伏；
2. 主升阶段超大单承接强；
3. 高位不是单纯拉抬，而是允许强换手；
4. 盘口承接结构没有直接崩。

### 7.2 为什么它高位没立刻崩
当前第一版判断是：
- 关键不只是主动买入；
- 更关键的是：
  - `超大单净额`
  - `oib`
  - 撤买 / 撤卖结构
  - 分歧日后的修复能力

也就是说它更像：
- **高位强换手 + 深承接**

而不是：
- 一波点火后一地鸡毛。
