# REQ-20260316-05-history-multiframe-monthly-pool-rollout

## 1. 基本信息
- 标题：新版历史多维历史月份扩展方案（固定池快照 + 逐月回补 + 逐月上线）
- 状态：ACTIVE
- 负责人：Codex
- 关联 Task ID：`CHG-20260316-05`
- 关联 CAP：`CAP-L2-HISTORY-FOUNDATION`, `CAP-HISTORY-30M`, `CAP-SANDBOX-REVIEW`

## 2. 背景与目标
- 当前新版历史多维在生产上已能稳定消费 `2026-03` 正式 L2 历史；
- 对 `2026-02` 及更早月份，最稳妥的首选路径**不是重跑 Windows 原始 ZIP**，而是优先复用已经在 sandbox V2 跑通的固定池 `review_5m_bars` 结果，按月提升到生产 `history_5m_l2 / history_daily_l2`；
- 因此本轮月度扩展策略收敛为：
  1. 先冻结一份固定股票池快照；
  2. 按月从 sandbox V2 提升到生产；
  3. 每个月完成后就直接在生产库生效，让新版历史多维立即可用。

## 3. 池子策略（冻结）
### 3.1 池子来源
- 优先使用 sandbox V2 已验证的固定池：
  - 沪深A
  - 排除 ST
  - 总市值 50~300 亿
- 当前池子规模约 `2788`。

### 3.2 红线
- 不要按“当前市值”每个月重算股票池；
- 必须先导出一份**固定快照**（`pool_snapshot_YYYYMMDD.symbols.txt`），后续历史月份补数都按这份名单跑；
- 这样可避免“池子随时间漂移，导致历史覆盖口径不一致”。

## 4. 回补顺序（推荐）
1. `2026-02`
2. `2026-01`
3. `2025-12`
4. `2025-11`
5. `2025-10`
6. 后续依次向前

理由：
- 越近的月份对产品价值越高；
- 逐月验收更容易定位问题；
- 任何一个月有问题，不会拖垮整个历史扩展计划。

## 5. 每月标准执行流程
### 5.1 月度准备
1. 冻结本月目标池快照；
2. 生成本月交易日清单；
3. 检查 sandbox V2 固定池结果是否已覆盖该月；
4. 生成月度 manifest：
   - 月份
   - 交易日列表
   - symbols 快照
   - 预计覆盖股票数

### 5.2 正式提升
- 对 `2026-02` 及更早月份，优先走 **sandbox V2 -> production** 提升链路：
  - 读取 `data/sandbox/review_v2/meta.db` 中冻结池；
  - 从 `symbols/{symbol}.db::review_5m_bars` 抽取目标月份；
  - 直接覆盖写入生产 `history_5m_l2 / history_daily_l2`；
- 对未来新增月份（例如 `2026-03` 之后的每日盘后正式增量）：
  - 仍由 Windows 正式回补链路负责；
  - 不与本卡的历史月份提升链路混用。

### 5.3 月度验收
每个月跑完后，至少要做四类检查：
1. **覆盖**：交易日是否全跑完，run 记录是否齐；
2. **失败**：显式失败数、空结果数是否都进入 failure/review queue；
3. **质量**：`quality_info` 数量与集中分布；
4. **抽样**：`sz000833` 等关键样本在 `5m / 30m / 1h / 日` 下抽样。

### 5.4 月度同步生产
- 本轮历史月份提升直接在**云端生产库**执行；
- 因 source 与 target 同在云端，无需再走“Windows 导出增量 -> 云端 merge”；
- 每个月提升完成后，新版历史多维即可直接消费该月数据，无需额外前端改版。

## 6. 执行原则
- 一次只处理一个月；
- 每个月完成后先验收，再同步生产；
- 同步生产通过后，再开始下一个月；
- 这样可以形成“月度交付节奏”，而不是积压到最后一次性上线。

## 7. 首批建议执行顺序
### Phase 1
- 冻结池快照；
- 先补 `2026-02`；
- 验证“sandbox V2 月度提升 -> 月度验收 -> 生产立即可用”整条链路。

### Phase 2
- 补 `2026-01`；
- 形成 `2026-01 ~ 2026-03` 连续覆盖。

### Phase 3
- 继续补 `2025-12 / 2025-11 / 2025-10 ...`；
- 逐月向前扩展。

## 8. 风险与回滚
- 风险：
  1. 若每个月都重算股票池，会导致历史覆盖口径漂移；
  2. 若整库覆盖生产，一旦单月结果异常，回滚成本高；
  3. 若不做月度验收直接批量上线，问题会延迟暴露。
- 回滚：
  - 按月份增量同步时，单月异常可直接回退该月 merge；
  - 不影响此前已验收月份。
