# REQ-20260316-05-history-multiframe-monthly-pool-rollout

## 1. 基本信息
- 标题：新版历史多维历史月份扩展方案（固定池快照 + 逐月回补 + 逐月上线）
- 状态：ACTIVE
- 负责人：Codex
- 关联 Task ID：`CHG-20260316-05`
- 关联 CAP：`CAP-L2-HISTORY-FOUNDATION`, `CAP-HISTORY-30M`, `CAP-SANDBOX-REVIEW`

## 2. 背景与目标
- 当前新版历史多维在生产上已能稳定消费 `2026-03` 正式 L2 历史；
- 若要继续把更多历史月份补到新版历史多维，最稳妥的策略不是“一次性全市场全历史猛跑”，而是：
  1. 先冻结一份固定股票池快照；
  2. 按月往前回补；
  3. 每个月完成后就同步生产，让新版历史多维立即可用。

## 3. 池子策略（冻结）
### 3.1 池子来源
- 优先使用 sandbox V2 已验证的固定池：
  - 沪深A
  - 排除 ST
  - 总市值 50~300 亿
- 当前池子规模约 `2788`。

### 3.2 红线
- 不要按“当前市值”每个月重算股票池；
- 必须先导出一份**固定快照**（`pool_snapshot_YYYYMMDD.symbols.txt`），后续历史月份补数都按这份名单跑；
- 这样可避免“池子随时间漂移，导致历史覆盖口径不一致”。

## 4. 回补顺序（推荐）
1. `2026-02`
2. `2026-01`
3. `2025-12`
4. `2025-11`
5. `2025-10`
6. 后续依次向前

理由：
- 越近的月份对产品价值越高；
- 逐月验收更容易定位问题；
- 任何一个月有问题，不会拖垮整个历史扩展计划。

## 5. 每月标准执行流程
### 5.1 月度准备
1. 冻结本月目标池快照；
2. 生成本月交易日清单；
3. 检查该月原始 `.7z` 是否齐全；
4. 生成月度 manifest：
   - 月份
   - 交易日列表
   - symbols 快照
   - 预计包数

### 5.2 正式回补
- 对该月每个交易日，继续走正式回补链路：
  - staging 准备
  - 分 shard
  - worker 执行 `l2_daily_backfill.py`
- 当前稳定主路径仍是：
  - 外部控制端 / 或未来 Windows 本机 OS 级控制器
  - `8` 个独立 worker

### 5.3 月度验收
每个月跑完后，至少要做四类检查：
1. **覆盖**：交易日是否全跑完，run 记录是否齐；
2. **失败**：显式失败数、空结果数是否都进入 failure/review queue；
3. **质量**：`quality_info` 数量与集中分布；
4. **抽样**：`sz000833` 等关键样本在 `5m / 30m / 1h / 日` 下抽样。

### 5.4 月度同步生产
- 不建议整库覆盖生产；
- 建议按月导出正式增量，再做云端 merge；
- 每个月同步完成后，新版历史多维即可直接消费该月数据，无需额外前端改版。

## 6. 执行原则
- 一次只处理一个月；
- 每个月完成后先验收，再同步生产；
- 同步生产通过后，再开始下一个月；
- 这样可以形成“月度交付节奏”，而不是积压到最后一次性上线。

## 7. 首批建议执行顺序
### Phase 1
- 冻结池快照；
- 先补 `2026-02`；
- 验证“月度回补 -> 月度验收 -> 月度上线”整条链路。

### Phase 2
- 补 `2026-01`；
- 形成 `2026-01 ~ 2026-03` 连续覆盖。

### Phase 3
- 继续补 `2025-12 / 2025-11 / 2025-10 ...`；
- 逐月向前扩展。

## 8. 风险与回滚
- 风险：
  1. 若每个月都重算股票池，会导致历史覆盖口径漂移；
  2. 若整库覆盖生产，一旦单月结果异常，回滚成本高；
  3. 若不做月度验收直接批量上线，问题会延迟暴露。
- 回滚：
  - 按月份增量同步时，单月异常可直接回退该月 merge；
  - 不影响此前已验收月份。
