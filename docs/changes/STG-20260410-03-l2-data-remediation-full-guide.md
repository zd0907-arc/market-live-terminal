## 4. 这次查出来的核心问题

## 4.1 问题一：原始挂单事件码不是单一格式
实际抽样结果已经确认：

- `sz000833`
  - `委托类型`：`0 / 1 / U`
- `sh603629`
  - `委托类型`：`A / D / S`

所以不能再把事件解释写死为单一版本。

### 含义
- 旧逻辑：
  - `0 = add`
  - `1/U = cancel`
- 新发现：
  - 至少还存在：
    - `A = add`
    - `D = cancel`
    - `S = 特殊状态行，应过滤`

### 影响
- 如果脚本只认 `0/1/U`
  - 那么一部分股票正常
  - 一部分股票会把挂单事件层全丢掉

---

## 4.2 问题二：Windows 正式跑数链路版本漂移
现在 Windows 跑数机上的正式链路，不是最新版本，而是停在较早阶段。

### 当前 Windows 链路状态
- `l2_daily_backfill.py`
  - 还停在“基础 `5m + daily`”阶段
  - 没有完整事件层解析
- `l2_history_db.py`
  - 还没完整进入事件因子 schema
  - 当前 Windows 正式库缺：
    - `total_volume`
    - `l2_add_*`
    - `l2_cancel_*`
    - `l2_cvd_delta`
    - `l2_oib_delta`

### 影响范围
Windows 正式库中现有：
- `2026-03-02 ~ 2026-03-13`
- `history_daily_l2 = 76,749`
- `history_5m_l2 = 3,461,334`
- `8,111` symbols

这一整批结果都只有：
- 成交级 L1/L2 主力/超大单
- 没有事件层

这不是单票 bug，而是**整批正式数据的能力缺口**。

---

## 4.3 问题三：本地与 Windows / 云端环境漂移
目前本地库不是权威全量正式库，而是少量样本 + 局部补数状态。

例如：
- 本地利通之前缺 `2026-03-09`
- 但 Windows 原始包存在，且 dry-run 能成功生成正式结果

这说明：
- 这不是源数据缺失
- 是环境之间没完全同步

---

## 4.4 问题四：磁盘与批处理方式必须受控
你提到 D 盘快满，这和现有运维策略是一致的：

### 当前冻结策略
- 原始包保留在：
  - `D:\MarketData`
- 解压与 staging 不在 D 盘长期堆积
- 统一临时目录：
  - `Z:\l2_stage\YYYYMMDD`

### 每日批处理语义
1. 每天一个压缩包
2. 解压到 staging
3. 切 `6~8` 个 worker（当前文档冻结为 `8` 个）
4. 跑完以后：
   - 汇总结果
   - 记录失败样本
   - 删除 staging
5. 原始 `.7z` 保留

### 注意
- 你口头记忆里可能是 E 盘
- 但仓库当前冻结文档是 `Z:\l2_stage`
- 如果机器实际可用的是 E 盘，建议：
  - 要么把 E 盘统一映射/约定成 `Z:\l2_stage`
  - 要么统一修改文档与脚本，避免双口径

## 8. 关于 2025 原始数据能不能删

## 8.1 先给结论
**可以考虑删，但不能现在立刻裸删。**

要先满足一个“榨干再删”的最低标准。

---

## 8.2 为什么不能现在直接删
因为 `2025-01 ~ 2026-02` 虽然没有挂单事件 raw，但它仍然是**成交级原始数据**。

如果以后你想做这些事情，raw 仍然有价值：
- 重新定义主力 / 超大单阈值
- 做更细的 `1m` / `tick` 统计
- 做母单集中度、拆单行为、成交节奏研究
- 研究你未来可能新增的别的策略

所以如果现在直接删 raw，相当于把“重新提取更细特征”的能力也删掉了。

---

## 8.3 什么叫“榨干到可以删”
我建议至少做到下面 3 层：

### 最低可删线（当前系统够用）
把 `2025-01 ~ 2026-02` 全部稳定沉淀成：
1. `history_5m_l2`
2. `history_daily_l2`
3. 覆盖一致、抽样通过

如果你的目标只是：
- 当前复盘
- 当前选股研究
- 当前波段策略

做到这里，删除 raw 的风险已经明显下降。

### 推荐可删线（更稳）
除了上面两张正式表，再额外沉淀一层**成交级研究特征快照**：
1. 日级母单特征
2. 5m 级买卖方向统计
3. 大单/超大单次数与金额分布
4. 若未来选股会用到的日级特征，提前固化成派生表

这样以后大部分研究都不需要再回 raw。

### 保守可删线（最稳）
在“推荐可删线”基础上：
1. 留一份月级 manifest / 校验报告
2. 至少保留少量代表性 raw 样本（月抽样）

这样后续如果对口径有争议，仍有少量法证样本可追。

---

## 8.4 我的建议
如果空间压力大，优先策略是：

1. **先不要删 `2026-03+` raw**
   - 这段还要重跑
   - 而且它有挂单事件层
2. **`2025-01 ~ 2026-02` 先做“价值榨干计划”**
   - 把正式 `5m + daily` 跑齐
   - 再评估是否加一层研究特征表
3. 满足后再删 raw

### 当前建议级别
- 对 `2025` raw：
  - **可以进入“准备删除”评估**
  - 但前提是先把正式派生结果做扎实
- 对 `2026-03+` raw：
  - **现阶段绝对不要删**

## 9. 批量任务执行原则（必须遵守）

## 9.1 不允许长期在 D 盘解压堆积
- 原始包在 `D:\MarketData`
- staging 在 `Z:\l2_stage`
- 每天跑完即删 staging

## 9.2 维持按日运行，不做全月一次性巨型解压
- 一个压缩包一个交易日
- 一天一天推进
- 每天处理完立即清 staging

## 9.3 并发维持 `6~8` worker
- 当前文档冻结为 `8 worker`
- 但如果磁盘 / IO / 机器状态吃紧，可临时降到 `6`
- 不建议为追求速度盲目上更高并发

## 9.4 正式流程要分两类
### A. 最近窗口（`2026-03+`）
- 目标：升级事件层
- 方法：按日重跑正式 L2

### B. 旧窗口（`2025-01 ~ 2026-02`）
- 目标：把成交级 raw 的价值榨干
- 方法：优先做正式 `5m + daily`，必要时补研究派生
