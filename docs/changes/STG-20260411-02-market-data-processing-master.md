## 3. 系统总目标

当前系统不是单一复盘工具，而是要长期支撑三件事：

### 3.1 盯盘
- 盘中 / 盘后看单票或自选池的资金变化；
- 看实时与历史的衔接；
- 看主力 / 超大单 / L1 / L2 强弱变化。

### 3.2 复盘
- 对单只股票做多周期回看；
- 看价格、资金、事件、盘口结构；
- 尤其是回答：
  - 为什么启动；
  - 为什么加速；
  - 为什么高位稳住；
  - 为什么冲高回落。

### 3.3 选股
- 盘后扫描全市场；
- 找“吸筹 -> 启动确认 -> 波段持有”的候选；
- 后续做历史回测与规则迭代。

所以数据层的目标不是只“存下来”，而是要支撑：
- **盯盘可读**
- **复盘可解释**
- **选股可研究**

## 4. 历史数据分段与真实能力边界

## 4.1 第一段：`2025-01 ~ 2026-02`
- 原始形态：**成交级 raw**
- 已确认：没有真实挂单事件 raw

### 这段能做什么
- 日线
- 5m
- 主力 / 超大单 `buy/sell/net`
- 成交级母单归因
- 买卖节奏
- 活跃度 / 参与度

### 这段不能做什么
- 真实挂单事件层
- `l2_add_*`
- `l2_cancel_*`
- `l2_oib_delta`

### 这段的正确定位
这段不是“没价值”，而是：
- **没有挂单层**
- 但**成交层价值很大**

所以这段的任务不是“补盘口事件”，而是：
- **把成交级价值榨干**

---

## 4.2 第二段：`2026-03+`
- 原始形态：**日包 7z**
- 文件包含：
  - `行情.csv`
  - `逐笔成交.csv`
  - `逐笔委托.csv`

### 这段能做什么
- 第一段所有能力
- 额外再做：
  - `total_volume`
  - `l2_add_buy_amount`
  - `l2_add_sell_amount`
  - `l2_cancel_buy_amount`
  - `l2_cancel_sell_amount`
  - `l2_cvd_delta`
  - `l2_oib_delta`
  - `quality_info`

### 这段的正确定位
这段才是后续：
- 高级复盘
- L2 确认型选股
- 出货风险识别

真正的核心原料。

所以：
- **2026-03+ raw 现阶段绝对不能删**

## 5. 当前查出来的数据问题

## 5.1 事件码不是单一格式
已抽样确认至少两套：

- 一套：`0 / 1 / U`
- 一套：`A / D / S`

当前应按下面兼容：
- `0 -> add`
- `1 -> cancel`
- `U -> cancel`
- `A -> add`
- `D -> cancel`
- `S -> ignore`

这不是单票问题，是原始包编码体系不统一的问题。

---

## 5.2 Windows 正式链路版本落后
当前 Windows 正式库不是“只差一个小补丁”，而是阶段性落后：

- 正式链路还停在基础 `5m + daily`
- 没完整进入事件层时代

直接影响：
- `2026-03-02 ~ 2026-03-13` 的正式全市场结果
- 虽然有 `history_5m_l2 / history_daily_l2`
- 但缺完整：
  - `l2_add_*`
  - `l2_cancel_*`
  - `l2_cvd_delta`
  - `l2_oib_delta`

---

## 5.3 环境漂移
现在有三层环境：
- 本地
- Windows 正式跑数机
- 云端

目前真实情况是：
- 有些数据 Windows raw 有；
- 但本地没同步；
- 云端有的地方只有 placeholder；
- 所以不能把“页面空”简单理解成“raw 没有”。

---

## 5.4 原始质量问题仍然存在
即使补完，仍会看到：
- `quality_info = OrderID 部分缺失，L2 数值可能偏小`

这类问题的处理原则是：
- 趋势 / 相对强弱可以看；
- 不把绝对值当成完全精确法证。

## 6. 老数据怎么处理（`2025-01 ~ 2026-02`）

## 6.1 目标
不是补挂单，而是把成交级价值尽量榨干，服务：
- 复盘
- 选股
- 删除 raw 前的价值固化

## 6.2 最低处理目标
先把整段稳定沉淀成：
- `history_5m_l2`
- `history_daily_l2`

这两张表要做到：
- 覆盖完整
- 口径一致
- 可按月审计

这是**最低可删线**。

## 6.3 推荐处理目标
除了正式 `5m + daily`，再额外沉淀一层**研究快照层**。

建议新增独立研究派生表/库，至少存：
- 日级母单特征
- 大单 / 超大单次数分布
- 大单 / 超大单金额分布
- 5m 级买卖方向统计
- 成交节奏特征
- 后续选股规则需要的 snapshot

这是**推荐可删线**。

## 6.4 为什么不能现在就删
因为一旦删了 raw：
- 后续若想重定义阈值；
- 重新提更细粒度特征；
- 做新的研究派生；

就很难回头。

所以当前结论：
- **2025-01 ~ 2026-02 可以进入“删除准备期”**
- **但现在还不能直接删**

## 10. Windows 批处理运行原则

## 10.1 空间原则
- 原始包：`D:\MarketData`
- staging：`Z:\l2_stage\YYYYMMDD`
- 每天跑完即删 staging
- 不在 D 盘长期堆解压文件

## 10.2 运行原则
- 按日处理
- 每日包处理完立刻清 staging
- 并发维持 `6~8 worker`
- 当前冻结优先按 `8 worker`

## 10.3 为什么不能粗暴全月重解压
因为你当前的约束不是 CPU，而是：
- 磁盘空间
- IO
- staging 清理
- 长任务稳定性

所以日级推进是对的。
