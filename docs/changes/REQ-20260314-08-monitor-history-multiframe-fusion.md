# REQ-20260314-08-monitor-history-multiframe-fusion

## 1. 基本信息
- 标题：盯盘页新版收敛为“当日分时 + 历史多维”双区，并规划 2026-03 L2 数据入库后的盯盘/复盘同源方案
- 状态：ACTIVE
- 负责人：Codex / 前后端 AI
- 关联 Task ID：`CHG-20260314-08`
- 关联 CAP：`CAP-REALTIME-FLOW`, `CAP-HISTORY-30M`, `CAP-L2-HISTORY-FOUNDATION`, `CAP-SANDBOX-REVIEW`
- 关联 STG：`STG-20260314-01`

## 2. 背景与目标
- 你已确认新版盯盘页不再沿用“当日分时 / 30分钟线 / 日线”三分结构。
- 新版目标改为：
  1. **保留一个“当日分时”区域**：专门盯当天盘中的实时情况；
  2. **新增一个“历史多维”区域**：统一承载 `5m / 30m / 1h / 日` 四种时间维度；
  3. 历史多维在视觉与数据口径上统一，不再把 30 分钟线与日线割裂成两个独立产品模块；
  4. 等 Windows `2026-03` L2 日包下载完成并清洗入库后，盯盘页与复盘页都要建立在更清晰的正式/实验分层之上。
- 本轮先做“文档先行”：把已完成事项收口，把未完成事项迁入待办，并冻结本期产品范围、数据方案、接口方向和实施顺序。

## 3. 方案与边界
- 做什么：
  - 冻结新版盯盘页的信息架构（IA）；
  - 冻结历史多维模块的数据分层与查询优先级；
  - 明确 `2026-03` L2 日包入库后盯盘页与复盘页的同源关系；
  - 明确 sandbox 在后续只保留实验/验证用途。
- 不做什么：
  - 本卡不直接落地新版盯盘页代码；
  - 本卡不直接启动复盘页并库实现；
  - 本卡不要求在 `2026-03` 全月数据尚未落库前，强行上线新版历史多维模块。

## 4. 产品结构决策（冻结）
### 4.1 新版盯盘页顶层结构
- 新版只保留两个主区域：
  1. `当日分时`
  2. `历史多维`
- 推荐切换方式：
  - 第一层：`旧版 / 新版`
  - 新版内部：`当日分时` 与 `历史多维` 两个主 Tab
  - 进入 `历史多维` 后，再用 segmented control 切换：`5m / 30m / 1h / 日`
- 理由：
  - 当日分时与历史分析的使用心智完全不同，应保持一级分离；
  - `30m` 与 `日线` 本质都是“历史聚合视角”，应归并到一个统一容器中；
  - `5m/30m/1h/日` 用同一模块切换，后续前端与后端都更容易同源化。

### 4.2 历史多维模块的展示目标
- 统一目标：同一套数据底座、同一套时间轴心智、不同粒度下切换观察。
- `5m / 30m / 1h`：
  - 主数据源来自正式 `history_5m_l2` 聚合；
  - 展示重点是价格区 + L1/L2 绝对资金对比；
  - 后续允许叠加净流、累计、锚点，但不在本期冻结具体副图数量。
- `日`：
  - 主数据源来自正式 `history_daily_l2`；
  - 维持已经验证过效果的“上方价格区 + 下方左超大/右主力双柱”主视觉。

## 5. 数据层方案（冻结）
### 5.1 正式历史层（finalized）
- 正式历史只认两张核心事实表：
  1. `history_5m_l2`
  2. `history_daily_l2`
- 规则：
  - `5m` 是所有历史多维聚合的统一底座；
  - `30m/1h` 不单独持久化，统一查询时从 `5m` 聚合；
  - `日` 直接查 `history_daily_l2`；
  - 正式表只存 **盘后结算后的 finalized 值**。

### 5.2 盘中预览层（preview）
- 为了支持新版“当日分时”和历史模块中对“今天”的展示，后续需要补充 preview 层：
  1. `realtime_5m_preview`
  2. `realtime_daily_preview`
- 规则：
  - preview 层只存盘中临时值；
  - 默认以 **L1 实时聚合** 为主，不伪造 L2；
  - 盘后正式 L2 到位后，由正式历史层覆盖“今天”的长期口径。

### 5.3 查询优先级（必须写死）
- `当日分时`
  - 今日交易时段：优先实时链路 / preview
  - 非当日历史回溯：优先 `history_5m_l2`
- `历史多维`
  - `5m/30m/1h`：
    - 历史日期：`history_5m_l2` 聚合
    - 今日：`realtime_5m_preview`（若需要纳入今天）
  - `日`：
    - 历史日期：`history_daily_l2`
    - 今日未结算：`realtime_daily_preview`（L1-only）
- 所有返回都必须显式带：
  - `source`
  - `is_finalized`
  - `preview_level`（若是 preview）
  - `fallback_used`

### 5.4 sandbox 与正式层边界
- `sandbox review v2` 继续保留：
  - 算法实验
  - 口径验证
  - 特殊样本研究
- 不再把 sandbox 当作新版盯盘页或未来正式复盘页的长期正式数据源。

## 6. 2026-03 L2 日包落库后的系统收口方案
### 6.1 盯盘页
- `当日分时`：继续实时优先；
- `历史多维`：统一正式底座；
- 当 `2026-03` 日包逐日完成清洗后，历史多维模块可逐步获得完整 `5m / 30m / 1h / 日` 正式值。

### 6.2 复盘页
- 目标不是立刻删除 sandbox，而是分两层：
  - **正式复盘视图**：后续逐步切到生产 `history_5m_l2/history_daily_l2`
  - **实验复盘视图**：继续保留 sandbox
- `2026-03` 数据到位后，应优先完成：
  1. 正式查询层字段对齐；
  2. 复盘页最小兼容接口；
  3. 再决定是否替换现有 sandbox 主页面。

## 9. 风险与回滚
- 风险：
  1. 若不把 `30m` 与 `日` 统一成历史多维模块，后续每次粒度扩展都要继续分裂前端结构；
  2. 若 preview 与 finalized 不分层，今天的数据会和正式历史混口径；
  3. 若 `2026-03` 正式数据未完成落库就强行做新版历史多维，容易出现“设计对了但页面没数”的假失败。
- 回滚：
  - 本卡是文档冻结，不改代码；
  - 前端后续即使实施失败，也可继续保留旧版页面作为安全回退入口。
