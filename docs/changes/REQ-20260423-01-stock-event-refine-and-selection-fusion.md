- 状态：ACTIVE
- 负责人：AI / Dong
- 关联 Task ID：`CHG-20260423-01`

## 2. 当前真实现状
- 已有 `stock_events` 事件底座、feed/coverage/audit 查询、单票 bundle 同步、watchlist 回补、选股页事件来源展示。
- 当前最大短板不是 schema，而是**可持续可用性**：
  1. 无 `TUSHARE_TOKEN` 时，公告还能走公共 fallback，但问答 / 新闻会明显降级；
  2. 选股链路还没有“候选票触发采集 → 事件理解 → 推荐解释”的闭环；
  3. 页面能看来源，但系统还没有输出结构化的“利好 / 利空 / 持续性 / 催化类型”。

## 3. 本轮目标（冻结）

### 3.1 P0：把事件层收成“按需触发”的稳定研究底座
- 不追求全市场常驻拉取；
- 以 `候选池 / 自选 / 重点研究票` 为主；
- 需要时能稳定、快速拿到最近事件。

### 3.2 P1：补齐无 Token 模式下的可用性信息
- 系统必须明确告诉上层：
  - 哪些来源可用；
  - 哪些来源当前因缺 token 被跳过；
  - 哪些模块有公共 fallback，哪些还没有。
- 禁止“静默为零”导致你误判成“这只票最近没新闻”。

### 3.3 P2：建立选股侧的触发式采集入口
- 当模型跑出候选票后：
  - 能对单票触发 `公告 + 问答 + 新闻` bundle；
  - 返回本次同步摘要、覆盖情况、最近事件流；
  - 供后续 AI 解释链路直接使用。

### 3.4 P3：建立事件理解层（下一阶段）
- 对单条/整组事件输出：
  - `event_polarity`：利好 / 利空 / 中性
  - `event_strength`：弱 / 中 / 强
  - `catalyst_type`：业绩 / 订单 / 监管 / 问答表态 / 题材 / 资本运作
  - `time_horizon`：盘后短催化 / 1~2 周 / 中期
  - `confidence / summary / evidence_url`
- 本轮先预留入口，不强求一次做完。

## 4. 推荐业务流程（冻结）
1. 每日盘后先跑现有选股模型，产出候选池；
2. 对候选池 TopN 触发单票事件包同步；
3. 对同步到的事件做结构化理解；
4. 把“资金 / 走势 / 情绪 / 事件”融合成最终推荐解释；
5. 选股详情页保留原文回看入口，方便你核对 AI 结论依据。

## 5. 本轮实施范围

### Phase 1（本次先做）
1. 新增“事件源能力视图”，明确 token / 公共 fallback 状态；
2. 新增“单票按需触发采集”接口，供候选票/研究页直接调用；
3. 审计结果里补“缺失是因为无数据，还是因为当前源不可用”；
4. 先把 `news` 补到公共 fallback 可用；
4. 文档同步更新成当前真实方案。

### Phase 2（紧接着做）
1. 补公共新闻源；
2. 评估公共问答源；
3. 统一无 token 模式的回补口径。

### Phase 3（后续）
1. 事件理解层；
2. 选股分数融合；
3. 复盘页/选股页完整引用链路。

## 6. 验收标准
- Given `2026-04-23 21:00` 未配置 `TUSHARE_TOKEN`，When 调用单票事件审计，Then 返回中必须明确指出哪些模块因源能力不足被跳过，而不是只返回 count=0。

- Given `2026-04-23 21:20` 某只票最近仅有公告无新闻，When 查看事件覆盖摘要，Then 系统必须展示“公告已覆盖、新闻未覆盖”，且可区分“真没采到”与“当前源不可用”。

## 7. 风险与边界
- 公共新闻/问答源若直接抓前端页面，稳定性会低于公告 fallback；
- 新闻多票归属仍然是高脏度问题，本轮不把“全市场多票实体识别”当成硬验收项；
- 事件理解层会引入 LLM 成本与 prompt 稳定性，本轮只先把采集与触发链路打稳。
