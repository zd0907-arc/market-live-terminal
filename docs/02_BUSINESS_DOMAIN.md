## 3. 能力卡片

### CAP-MKT-TIME（交易日与交易时段展示状态机）

1. **业务目标**
- 在任意时刻都展示正确“交易状态 + 展示日期（display_date）”，避免误回溯、误前推、误复制。

2. **业务规则（强约束）**
- **时间基准**：北京时间（Asia/Shanghai）。
- **交易日状态机**：
  - `09:15-11:30`：上午交易态（实时模式，`display_date=当日`）
  - `11:30-13:00`：午休停盘态（非撮合，但仍属当日展示，`display_date=当日`）
  - `13:00-15:00`：下午交易态（实时模式，`display_date=当日`）
  - `15:00-24:00`：收盘态（当日展示，不切到次日）
  - `00:00-09:15`：盘前态（展示最近交易日）
- **非交易日**：全天展示最近交易日。
- **数据来源优先级**：交易日判断优先使用交易日历；失败时允许保守降级（不误判为交易进行中）。
- **禁止行为（红线）**：
  - 禁止将“前一交易日数据”复制成“自然日当天数据”；
  - 禁止非交易日写入“当天交易数据”。

3. **实现摘要（当前方案）**
- 后端 `MarketClock` 统一产出 `display_date` 与交易时段判断，前端按 `display_date` 判定是否回溯标签。
- 交易日历由后端集中维护，失败时走保守策略，避免“假实时”。

4. **验收案例（Given / When / Then）**
- **正常**：Given `2026-03-09 10:00`，When 打开实时页，Then `display_date=2026-03-09` 且非回溯模式。
- **边界**：Given `2026-03-09 12:00`（午休），When 打开实时页，Then 仍显示 `2026-03-09`，且不应被标记为历史回溯。
- **异常防线**：Given `2026-03-10 00:30`，When 请求实时接口，Then `display_date` 必须是最近交易日，且不得出现“复制成 2026-03-10 的盘中数据”。

5. **可观测性与告警**
- 观测点：`/api/realtime/dashboard` 返回的 `display_date`、前端“回溯模式”标签、后端时间判定日志。
- 异常信号：交易日盘中出现回溯标签，或非交易日出现当天交易数据。

6. **变更记录（任务ID）**
- `CHG-20260307-01`：修复凌晨/周末误写当日问题（避免前一日复制成今天）。
- `CHG-20260309-01`：补强时区口径一致性，修复盘中误回溯显示。

### CAP-REALTIME-FLOW（主力动态实时数据链路）

1. **业务目标**
- 在交易时段稳定展示“主力动态”（分钟级曲线 + 最新逐笔），即使用户未手动刷新也能持续有数据。

2. **业务规则（强约束）**
- **主力定义**：
  - 超大单：单笔成交金额 `>= 1,000,000`
  - 大单：单笔成交金额 `>= 200,000` 且 `< 1,000,000`
  - 主力资金 = 大单 + 超大单
- **口径红线**：历史与实时统一使用“单笔金额阈值判定”，禁止因历史有订单号而单独启用拆单还原。
- **数据来源优先级**：
  1. 当日 `trade_ticks`（本地库）
  2. 本地采集任务拉取并覆盖写入
- **成交方向兼容口径**：
  - 必须兼容 `买盘/卖盘/中性盘`、`buy/sell/neutral`、`B/S/M` 三类来源标记；
  - 对外统一标准化为 `buy/sell/neutral`，未知标记默认降级为 `neutral` 并打诊断日志。
- **禁止行为（红线）**：
  - 当日 `trade_ticks=0` 时静默显示“正常实时”；
  - 改阈值后不记录影响范围。

3. **实现摘要（当前方案）**
- 实时链路：采集（AkShare/Windows） -> `trade_ticks` 入库 -> 分钟聚合 -> `/api/realtime/dashboard` 输出。
- 聚合使用同一阈值配置，输出 `chart_data`、`cumulative_data`、`latest_ticks`。
- 分时与历史聚合共享成交方向归一化模块，避免因源字段枚举差异导致主力资金线归零。

4. **验收案例（Given / When / Then）**
- **正常**：Given `2026-03-09 10:30` 且 watchlist 已采集，When 请求 `/api/realtime/dashboard?symbol=sz000833`，Then `chart_data` 非空且 `latest_ticks` 至少 1 条。
- **边界**：Given `2026-03-09 23:31` 当日 ticks 的 `type` 同时出现 `B/S/M` 与 `买盘/卖盘`，When 请求同一接口，Then 主力资金字段应保持非零且 `latest_ticks.type` 统一为 `buy/sell/neutral`。
- **边界**：Given 首次启动且尚未有当日 ticks，When 请求接口，Then 返回空数据并有可诊断信号（日志/监控），不得伪造曲线。
- **异常防线**：Given 采集器异常中断，When 连续请求接口，Then 能明确观测到 `trade_ticks` 未增长并触发排查路径。

5. **可观测性与告警**
- 观测点：`trade_ticks` 当日行数、`/api/realtime/dashboard` 数据长度、采集任务日志。
- 快速诊断：`trade_ticks(date=today)=0` 即判定为数据底座缺失，不先责怪前端图表。
- 附加诊断：若 `latest_ticks.type` 出现未知枚举，需记录采样并评估是否补充归一化规则。

6. **变更记录（任务ID）**
- `CHG-20260307-02`：增强无人查看场景下的落数策略（周期全量轮扫 + 收盘重试）。
- `CHG-20260309-02`：本地快速补数恢复（watchlist 11 只股票当日 ticks 回填）。
- `CHG-20260309-09`：修复分时资金流方向枚举不兼容（`B/S/M` 与中文/英文口径统一归一化）。
- `CHG-20260310-01`：修复生产当日分时无数据（Windows 依赖缺失 + ingest 目标地址错误），恢复持续上报与实时分时展示。

### CAP-HISTORY-30M（30分钟历史趋势与口径）

1. **业务目标**
- 30分钟线稳定反映主力资金趋势，日内桶定义一致，历史与实时可解释。

2. **业务规则（强约束）**
- **标准桶**：每交易日固定 8 个 30m 桶（`09:30,10:00,10:30,11:00,13:00,13:30,14:00,14:30`）。
- **数据来源优先级**：
  1. `history_30m`（主力流向）
  2. 必要时由 ticks 重算（同口径）
- **禁止行为（红线）**：
  - 禁止把仅有OHLC、资金流为0的占位K线混写为真实资金流记录；
  - 禁止 8桶/10桶混口径对外展示。

3. **实现摘要（当前方案）**
- ETL、后端聚合、API读取统一 8 桶映射。
- API 层过滤非标准桶，backfill 默认禁用“零流入占位写入”路径。

4. **验收案例（Given / When / Then）**
- **正常**：Given `2026-02-13`，When 请求 `/api/history/trend?symbol=sz000833&days=20`，Then 当日应只出现 8 个标准时间桶。
- **边界**：Given 历史源存在 `11:30/15:00` 额外桶，When 入库或读取，Then 必须被归并/过滤，不得直接外显。
- **异常防线**：Given 出现“OHLC 非0但资金流全0”大面积记录，When 审计，Then 标记为污染窗口并进入重建流程。

5. **可观测性与告警**
- 观测点：`bar_count/day`、`zero_flow_ohlc_count`、关键票（如 `sz000833`）抽样比对。
- 告警条件：单日桶数不等于 8，或占位0流入比例异常升高。

6. **变更记录（任务ID）**

- `CHG-20260308-01`：Windows ETL 按新口径重建 `202602` 数据（待回传merge完成最终闭环）。

### CAP-WIN-PIPELINE（Windows离线节点与云端注入）

1. **业务目标**
- 利用 Windows 节点承担离线重建与盘中采集，向云端被动注入数据，保证在云端禁外采约束下系统可持续运行。

2. **业务规则（强约束）**
- **节点职责**：
  - Windows：采集/ETL 产出；
  - 云端：只接收 ingest 与提供查询；
  - Mac：开发与运维中控。
- **路径约定**：Windows 统一目录 `D:\market-live-terminal`。
- **数据流红线**：
  - 云端默认不主动外采（`ENABLE_CLOUD_COLLECTOR=false` for production）；
  - Windows 不作为 Git 主仓操作节点。
- **禁止行为（红线）**：
  - 在连接不稳定条件下进行无校验大文件覆盖；
  - 未核对文件完整性即执行云端 merge。

3. **实现摘要（当前方案）**
- 通过 Tailscale + SSH/SCP 远程驱动 Windows 脚本与 ETL；产物回传后在云端执行 merge。
- 目前已实现免密登录，但连接稳定性（active但不通）仍是短板。

4. **验收案例（Given / When / Then）**
- **正常**：Given Windows在线且SSH可达，When 执行 `sync_to_windows.sh`，Then 脚本同步成功且可远程触发 ETL。
- **边界**：Given 95MB 历史库通过 DERP 回传，When 传输完成，Then 本地文件大小/校验必须与源文件一致。
- **异常防线**：Given `tailscale status` 显示 active 但 `ping/ssh` 超时，When 触发流程，Then 任务必须进入阻塞待办而不是盲目继续。

5. **可观测性与告警**
- 观测点：`tailscale ping` 成功率、SSH连接成功率、大文件回传时延、Windows `sshd` 服务状态。
- 告警条件：连续连接失败、ETL产物回传中断、merge前校验失败。

- `CHG-20260308-02`：Windows SSH 免密打通与脚本同步恢复。
- `CHG-20260309-03`：确认 ETL 已重建完成但回传/merge受连接抖动阻塞（登记 `T-002/T-003`）。

### CAP-SANDBOX-REVIEW（L1/L2 复盘沙盒验证闭环）

1. **业务目标**
- 在完全隔离的沙盒链路中，复盘同一时间轴下 L1 与 L2 资金行为差异，并验证“母单还原”在妖股阶段的解释力。

2. **业务规则（强约束）**
- **隔离红线**：
  - 只读源 CSV/ZIP，不写生产 `market_data.db`；
  - API 仅新增 `/api/sandbox/*`；
  - 前端仅新增 `/sandbox-review` 页面与入口，不改现有业务图表口径。
  - 非生产发布策略固定为 `单分支 + Draft PR`，在数据就绪前禁止合并主干与生产部署。
- **时间粒度**：5 分钟切片（非交易时段不补零）。
- **L1 口径**：
  - 主力阈值：单笔 `amount >= 200,000`；
  - 超大单阈值：单笔 `amount >= 1,000,000`；
  - 输出绝对买入、绝对卖出、净流入。
- **L2 口径（母单还原）**：
  - 先按 `BuyOrderID`、`SaleOrderID` 日内汇总金额并打标；
  - 逐笔记账时，买方与卖方独立计入各自 buy/sell 侧；
  - 主力对倒不做过滤，允许上下柱同时放大，净流入单独反映方向性。
  - **严格模式**：缺失 `BuyOrderID/SaleOrderID` 列的源文件必须直接失败，禁止降级近似 L2。
- **金额口径**：
  - 优先使用源字段 `Amount/成交额`；
  - 缺失时依据 `Volume` 单位自动判定“股/手”后计算金额，并输出检测日志。
- **数据质量防线**：
  - 查询接口需检测“整日重复分时”异常（同一日内所有 5m OHLC 与资金字段与另一交易日完全一致）；
  - 默认剔除后出现的重复日，并在接口 `message` 返回异常日期对（如 `2026-02-23≈2026-02-11`）。

3. **实现摘要（当前方案）**
- 新增 sandbox ETL 脚本，输出独立库 `sandbox_review.db`（表：`review_5m_bars`）。
- 新增沙盒查询接口 `/api/sandbox/review_data`，返回标准 `APIResponse`。
- 新增前端页面 `/sandbox-review`：主图 K 线 + 资金多副图（绝对买卖、净流、净流比、锚点累计）。
- 页面交互：双端滑块横向拖动 + 快捷窗口（1/3/5/20/60日/全部），不使用滚轮缩放。
- 动态聚合：自动规则 `1日=5m, 3/5日=15m, 20日=60m, 60日/全部=1d`，并提供手动粒度覆盖（`5m/15m/30m/60m/1d`）。
- 前端可信度策略：Fail-Closed；接口失败或空数据时不回退本地预置数据，只展示错误/空状态。
- 统计验证扩展：新增相关性模块（`l1/l2 net` 与 `price_return/next_price_return` 的 Pearson）；活跃度口径使用 `total_amount`（该股票5分钟总成交额）并支持 `l2_activity_ratio>30%` 条件过滤。
- 锚点累计模式：点击 K 线设锚点后，累计区按 `主力L2/主力L1/超大L2/超大L1` 四图独立展示；每图采用“单累计曲线 + 正负面积分色”。
- 采用“先单日试跑，再全量刷数”的执行节奏。

4. **验收案例（Given / When / Then）**
- **正常**：Given `2026-03-10 20:00` 运行单日试跑，When ETL 完成，Then `review_5m_bars` 包含目标股票当日 5m 数据且有 L1/L2 buy/sell/net 字段。
- **正常**：Given `2026-03-10 20:20` 请求 `/api/sandbox/review_data`，When 参数合法，Then 返回 `code=200` 且时间升序数组。
- **边界**：Given 某个源文件缺失 `BuyOrderID/SaleOrderID`，When 启动 ETL 严格模式，Then 任务必须失败并输出缺失文件清单。
- **异常防线**：Given 误配置输出路径指向 `market_data.db`，When 启动 ETL，Then 任务应拒绝执行并报错。

5. **可观测性与告警**
- 观测点：ETL 金额单位判定日志、`review_5m_bars` 行数、接口返回长度、前端六图联动表现（同一时间轴）。
- 告警条件：出现写入生产库风险、金额单位无法判定、单日试跑无有效 bars。

## 4. 附录：统一术语

- **display_date**：前端展示日期，不等于自然日，必须服从 `CAP-MKT-TIME` 状态机。
- **回溯模式**：`display_date != 当日(北京时间)` 时的历史展示态。
- **污染窗口**：历史数据中存在明显占位或口径混杂、导致分析不可用的日期区间。
