## 3. 能力卡片

### CAP-MKT-TIME（交易日与交易时段展示状态机）

1. **业务目标**
- 在任意时刻都展示正确“交易状态 + 展示日期（display_date）”，避免误回溯、误前推、误复制。

2. **业务规则（强约束）**
- **时间基准**：北京时间（Asia/Shanghai）。
- **交易日状态机**：
  - `09:15-11:30`：上午交易态（实时模式，`display_date=当日`）
  - `11:30-13:00`：午休停盘态（非撮合，但仍属当日展示，`display_date=当日`）
  - `13:00-15:00`：下午交易态（实时模式，`display_date=当日`）
  - `15:00-24:00`：收盘态 / 盘后复盘（当日展示，不切到次日）
  - `00:00-09:15`：**隔夜复盘态**（继续按复盘语义展示上一交易日，不显示“盘前未开盘”）
- **非交易日**：全天展示最近交易日。
- **前后端展示约束**：
  - 前端不得再自行仅凭浏览器本机时间输出最终“实时交易中/回溯模式”文案；
  - 最终展示必须以后端返回的 `market_status / market_status_label / default_display_scope / view_mode` 为准；
  - 在后端数据尚未返回前，前端允许展示“临时状态”，但临时状态语义必须与本 CAP 一致，不得出现盘后误报为“实时交易中”。
- **数据来源优先级**：交易日判断优先使用交易日历；失败时允许保守降级（不误判为交易进行中）。
- **禁止行为（红线）**：
  - 禁止将“前一交易日数据”复制成“自然日当天数据”；
  - 禁止非交易日写入“当天交易数据”；
  - 禁止在交易日盘后仍把状态显示成“实时交易中”。

3. **实现摘要（当前方案）**
- 后端 `MarketClock` 统一产出 `market_context`，明确给出：
  - `market_status / market_status_label`
  - `default_display_date`
  - `default_display_scope`
  - `should_use_realtime_path`
- 前端顶部状态 badge 不再自己猜最终状态，而是以后端权威字段为准；当数据尚未返回时，只允许显示与本 CAP 一致的临时状态。
- 交易日历由后端集中维护，失败时走保守策略，避免“假实时”。
- `/api/realtime/dashboard` 在“自然日当天且为交易日”时才允许走实时 ticks 聚合；若 `display_date` 回溯到上一交易日（周末/节假日/盘前），则优先走 `history_1m` 静态回放；若该日 `history_1m` 未准备好但 `trade_ticks` 已存在，则回退为“按该日 ticks 聚合”，避免周末页面空白。
- 对于“交易日当天但不在盘中（午休 / 盘后）”的场景，默认仍展示**今天**的数据，只是默认不再走“实时拉取路径”。

4. **验收案例（Given / When / Then）**
- **正常**：Given `2026-03-09 10:00`，When 打开实时页，Then `display_date=2026-03-09` 且非回溯模式。
- **边界**：Given `2026-03-09 12:00`（午休），When 打开实时页，Then 仍显示 `2026-03-09`，且不应被标记为历史回溯。
- **异常防线**：Given `2026-03-10 00:30`，When 请求实时接口，Then `display_date` 必须是最近交易日，且不得出现“复制成 2026-03-10 的盘中数据”。

5. **可观测性与告警**
- 观测点：`/api/realtime/dashboard` 返回的 `display_date`、前端“回溯模式”标签、后端时间判定日志。
- 异常信号：交易日盘中出现回溯标签，或非交易日出现当天交易数据。

6. **变更记录（任务ID）**
- `CHG-20260307-01`：修复凌晨/周末误写当日问题（避免前一日复制成今天）。
- `CHG-20260309-01`：补强时区口径一致性，修复盘中误回溯显示。
- `CHG-20260314-01`：修复周末“当日分时”空白问题；非交易日回溯显示上一交易日时，改走 `history_1m` 静态回放而非实时 ticks 聚合。
- `CHG-20260318-02`：把盯盘页市场状态收口为 `盘前 / 盘中 / 午间休市 / 盘后 / 休盘日` 五态；交易日盘后默认展示今日数据，前端最终状态以后端权威状态字段为准。
- `CHG-20260319-02`：将交易日 `00:00-09:15` 的展示语义从“盘前未开盘”改为“隔夜复盘”，默认继续展示上一交易日复盘数据，避免午夜后错误切到盘前心智。

### CAP-REALTIME-FLOW（主力动态实时数据链路）

1. **业务目标**
- 在交易时段稳定展示“主力动态”（分钟级曲线 + 最新逐笔），即使用户未手动刷新也能持续有数据。

2. **业务规则（强约束）**
- **主力定义**：
  - 超大单：单笔成交金额 `>= 1,000,000`
  - 大单：单笔成交金额 `>= 200,000` 且 `< 1,000,000`
  - 主力资金 = 大单 + 超大单
- **已实施变更点（`CHG-20260319-01`）**：
  - 当前当日分时页内的 `主力动态` 与 `资金博弈分析` 已统一收敛到 **5m** 标准时间轴；
  - 页面模式固定为：
    1. `intraday_l1_only`
    2. `postclose_dual_track`
    3. `historical_dual_track`
  - 当 `query_date=当天交易日` 且 `finalized L2` 已到位时，页面自动从 L1 单轨切为 L1/L2 双轨，不提供手动开关；
  - 当日分时页现统一消费页面级接口 `/api/realtime/intraday_fusion`，旧 `/api/sentiment*` 已退出资金博弈正式主路径；
  - 盘中模式只允许展示 L1 5m，不触发撤单/隐藏单文字信号。
- **前端刷新状态机**：
  - 仅保留二态：`盯盘开=5秒`、`盯盘关=30秒`；
  - 周期刷新必须采用“静默刷新”语义：请求失败时保留上一版成功数据，禁止先清空图表再等待回填。
- **活跃心跳分层**：
  - 实时页只对“当前正在查看的股票”注册心跳；
  - 心跳模式仅有 `focus` 与 `warm` 两层；
  - 历史日期查看（手动选择过去交易日）禁止继续注册实时心跳。
- **Windows 抓取节奏**：
  - `focus_symbols`：快照 `3秒`、tick `5秒`；
  - `warm_symbols`：tick `30秒`；
  - watchlist 兜底轮扫维持 `15分钟`；
  - 本轮明确不引入“最近查看股票”中频轮扫队列。
- **口径红线**：历史与实时统一使用“单笔金额阈值判定”，禁止因历史有订单号而单独启用拆单还原。
- **数据来源优先级**：
  1. 当日 `trade_ticks`（本地库）
  2. 本地采集任务拉取并覆盖写入
  3. 若用户在交易日当天（尤其盘后复盘）查看一只本地尚无分时的新股票，允许由 `/api/realtime/dashboard` 触发**按需补抓当天 full-day ticks**，再写入本地并聚合展示；
  4. 该“按需补抓”只适用于“今天”的查看语义，不得越权把历史日期伪造成今天。
- **成交方向兼容口径**：
  - 必须兼容 `买盘/卖盘/中性盘`、`buy/sell/neutral`、`B/S/M` 三类来源标记；
  - 对外统一标准化为 `buy/sell/neutral`，未知标记默认降级为 `neutral` 并打诊断日志。
- **禁止行为（红线）**：
  - 当日 `trade_ticks=0` 时静默显示“正常实时”；
  - 改阈值后不记录影响范围。

3. **实现摘要（当前方案）**
- 实时链路：采集（AkShare/Windows） -> `trade_ticks` 入库 -> 分钟聚合 -> `/api/realtime/dashboard` 输出。
- 聚合使用同一阈值配置，输出 `chart_data`、`cumulative_data`、`latest_ticks`。
- 分时与历史聚合共享成交方向归一化模块，避免因源字段枚举差异导致主力资金线归零。
- 实时页心跳已升级为 `focus/warm` 分层；Windows crawler 按 `focus=5秒 tick / warm=30秒 tick / watchlist=15分钟兜底` 执行。
- 前端实时页改为二态盯盘按钮，并在轮询时保留旧数据，避免“闪空白”。
- 当用户查看“今天”的分时但本地尚无该股票数据时，后端允许按需抓取 full-day ticks 并写回本地，再补 `history_1m` 聚合，避免盘后查看新股票时一直空白。
- 当 today payload 已存在但个别股票停在 `14:45` / `14:50` 等陈旧时间点时，后端仍保留“页面访问触发的 stale rehydrate”；同时从 `2026-03-20` 起，云端 scheduler 会在交易日盘后 `15:02 / 15:07 / 15:12 / 15:17` 主动扫描自选股，若最新 tick 时间 `< 14:55:00`，则自动补抓、覆盖写回并刷新 `history_1m + realtime_5m_preview`。

4. **验收案例（Given / When / Then）**
- **正常**：Given `2026-03-09 10:30` 且 watchlist 已采集，When 请求 `/api/realtime/dashboard?symbol=sz000833`，Then `chart_data` 非空且 `latest_ticks` 至少 1 条。
- **边界**：Given `2026-03-09 23:31` 当日 ticks 的 `type` 同时出现 `B/S/M` 与 `买盘/卖盘`，When 请求同一接口，Then 主力资金字段应保持非零且 `latest_ticks.type` 统一为 `buy/sell/neutral`。
- **边界**：Given 首次启动且尚未有当日 ticks，When 请求接口，Then 返回空数据并有可诊断信号（日志/监控），不得伪造曲线。
- **异常防线**：Given 采集器异常中断，When 连续请求接口，Then 能明确观测到 `trade_ticks` 未增长并触发排查路径。

5. **可观测性与告警**
- 观测点：`trade_ticks` 当日行数、`/api/realtime/dashboard` 数据长度、采集任务日志。
- 快速诊断：`trade_ticks(date=today)=0` 即判定为数据底座缺失，不先责怪前端图表。
- 快速诊断补充：若 `trade_ticks(date=today)` 未增长，优先检查 Windows 计划任务 `ZhangDataLiveCrawler` 的“最后运行时间 / 当前是否在跑”；`2026-03-16` 已验证这类故障会直接导致生产实时 K 线空白。
- 附加诊断：若 `latest_ticks.type` 出现未知枚举，需记录采样并评估是否补充归一化规则。

6. **变更记录（任务ID）**
- `CHG-20260307-02`：增强无人查看场景下的落数策略（周期全量轮扫 + 收盘重试）。
- `CHG-20260309-02`：本地快速补数恢复（watchlist 11 只股票当日 ticks 回填）。
- `CHG-20260309-09`：修复分时资金流方向枚举不兼容（`B/S/M` 与中文/英文口径统一归一化）。
- `CHG-20260310-01`：修复生产当日分时无数据（Windows 依赖缺失 + ingest 目标地址错误），恢复持续上报与实时分时展示。
- `CHG-20260312-02`：盯盘三态收敛为二态（5s/30s），实时页改为静默刷新，并将 heartbeat / crawler 升级为 `focus/warm` 分层抓取。
- `CHG-20260318-02`：新增“交易日当天但盘后查看新股票”按需补抓与聚合逻辑，并把实时页状态展示改为以后端权威状态为准。
- `CHG-20260320-01`：新增盘后主动自愈扫盘，不再只依赖用户打开页面时才触发 today stale rehydrate。

### CAP-RETAIL-SENTIMENT（散户一致性观察 / 股吧舆情）

1. **业务目标**
- 提供一个基于统一舆情事件流的**散户一致性观察模块**，用于辅助判断“有没有人在讨论、最近是更热还是更冷、价格是否与讨论同步/背离”；
- 该能力是首页辅助观察模块，不承担盘中高频交易信号职责。

2. **业务规则（强约束）**
- **模块定位冻结（`CHG-20260322-01` / `CHG-20260323-01`）**：
  - 对外定位为 `散户一致性观察`，而不是“实时情绪交易信号”；
  - 首页主视图优先表达 `热度 -> 相对热度 -> 价格联动 -> AI日评`；
  - 关键词/方向分类不再作为首页主图主结论。
- **正式数据模型冻结**：
  - 正式舆情底座为 `sentiment_events`；
  - 事件流字段至少包含 `source / event_type / thread_id / parent_id / content / pub_time / interaction metrics`；
  - 旧 `sentiment_comments` 仅保留为兼容层与历史回填来源，不再是首页正式主消费表。
- **读接口红线**：
  - `GET /api/sentiment/dashboard/{symbol}` 禁止同步调用 LLM；
  - 首页正式主链路改为 `GET /api/sentiment/overview|heat_trend|feed|daily_scores`；
  - 页面不得把“页面刷新时间”当作“数据更新时间”。
- **首页窗口与主图冻结（`CHG-20260324-01` 实际收口）**：
  - 首页正式窗口为 `5D / 20D / 60D`，默认 `20D`；
  - 左侧主图固定为上下两层：
    - 上层：`价格 + AI情绪分`
    - 下层：`raw_heat` 柱 + `relative_heat_index` 线 + `AI一致性/温度` 线；
  - 图内只展示数值因子，不直接贴叙事类文字标签；
  - `raw_heat` 与 `relative_heat_index` 的口径以 `retail_sentiment.py` 中 V3 实现为准。
- **来源与原文流冻结（`CHG-20260324-01` 实际收口）**：
  - 当前正式源只保留 `股吧`；
  - 右侧原文流必须保留全文，不截断为摘要；
  - 当前正式页面不再展示多来源切换 Tab。
- **AI 日评策略**：
  - AI 解读已从“占位”升级为**星标股日级缓存结果**；
  - 输出 `情绪得分 / 一致性 / 温度 / 风险标签 / 摘要`；
  - 首页读请求只读缓存，不得临时同步调模。
- **覆盖边界**：
  - 当前正式可见数据仍以 `股吧主帖 + 旧评论兼容回填` 为主；
  - `reply` 事件与更完整事件关系链仍属后续补齐项；
  - 当前抓取/补数重点围绕 `watchlist(星标股)`，未扩到独立 `focus_pool / hot_pool`。

3. **实现摘要（当前方案）**
- 截至 `2026-03-24`，该能力的当前真实状态为：
  - 后端已落地 `sentiment_events`，并支持旧 `sentiment_comments` 懒回填；
  - 首页正式主链路已切到 `overview / heat_trend / feed / daily_scores`；
  - 首页已重构为 `左图 + 中AI + 右原文流`；
  - 星标股支持：新加即回补 `20D -> 60D`、日常定时抓取、日级 AI 评分缓存；
  - 当前正式源只保留 `股吧`；
  - 旧 `dashboard / trend / comments / keywords` 继续保留，供兼容与历史验证使用。

4. **验收案例（Given / When / Then）**
- **正常**：Given `2026-03-24 20:30`，When 首页打开 `sz000833` 并切到 `5D`，Then 左侧主图显示价格 + 事件数柱 + 相对热度线，右侧原文流可查看完整事件正文。
- **边界**：Given `2026-03-24 20:35` 某交易日价格有数据但该桶没有任何事件，When 查看 `heat_trend`，Then 该桶必须表现为 gap，而不是“热度归零平台”。
- **异常防线**：Given `2026-03-24 20:40` 请求 `GET /api/sentiment/overview/{symbol}`，When 后端返回 overview，Then 不得触发任何同步 LLM 调用。

5. **可观测性与告警**
- 观测点：
  - `latest_event_time`
  - `window total_events / post_count / reply_count`
  - `relative_heat_index`
  - 各来源事件数与未覆盖股票占比
- 告警条件：
  - 读接口仍触发 LLM；
  - `heat_trend` 无法区分 gap / 0；

### CAP-SANDBOX-REVIEW（L1/L2 复盘沙盒验证闭环）

1. **业务目标**
- 在完全隔离的沙盒链路中，复盘同一时间轴下 L1 与 L2 资金行为差异，并验证“母单还原”在妖股阶段的解释力。

2. **业务规则（强约束）**
- **隔离红线**：
  - 只读源 CSV/ZIP，不写生产 `market_data.db`；
  - API 仅新增 `/api/sandbox/*`；
  - 当前前端正式复盘 canonical 入口为 `/review`；`/sandbox-review` 仅保留兼容跳转语义。
  - 非生产发布策略固定为 `单分支 + Draft PR`，在数据就绪前禁止合并主干与生产部署。
- **时间粒度**：5 分钟切片（非交易时段不补零）。
- **L1 口径**：
  - 主力阈值：单笔 `amount >= 200,000`；
  - 超大单阈值：单笔 `amount >= 1,000,000`；
  - 输出绝对买入、绝对卖出、净流入。
- **L2 口径（母单还原）**：
  - 先按 `BuyOrderID`、`SaleOrderID` 日内汇总金额并打标；
  - 逐笔记账时，买方与卖方独立计入各自 buy/sell 侧；
  - 主力对倒不做过滤，允许上下柱同时放大，净流入单独反映方向性。
  - **严格模式**：缺失 `BuyOrderID/SaleOrderID` 列的源文件必须直接失败，禁止降级近似 L2。
- **金额口径**：
  - 优先使用源字段 `Amount/成交额`；
  - 缺失时依据 `Volume` 单位自动判定“股/手”后计算金额，并输出检测日志。
- **数据质量防线**：
  - 查询接口需检测“整日重复分时”异常（同一日内所有 5m OHLC 与资金字段与另一交易日完全一致）；
  - 默认剔除后出现的重复日，并在接口 `message` 返回异常日期对（如 `2026-02-23≈2026-02-11`）。

- **一期研究口径冻结**：
  - 盘后研究、次日及后续波段参考；
  - 默认候选池为 `Top10 breakout`；
  - 默认固定持有期回测为 `5/10/20/40`；
  - 入场口径固定为“信号日下一可用交易日收盘”；
  - 同时输出“固定持有结果 + 持有窗口内最高机会结果”。
- **L2 使用冻结**：
  - `2025-01 ~ 2026-02` 以日级资金为主；
  - `2026-03+` 若正式库已有可用 L2，则作为确认增强；
  - 若正式库未完整 merge 订单事件因子，新模块只能走弱化 L2 确认，不得因此修改旧模块读口径。
